# backend — серверная логика (npm workspace-пакет)

`@portfolio/backend` — не отдельный сервис, а npm-пакет верхнего уровня,
который `web/` (Next.js) подключает как обычную зависимость через npm
workspaces. Рантайм всё равно один: один Next.js-процесс, один systemd-сервис
на VPS (см. журнал решений в `web/README.md`, записи про архитектуру backend
и про этот же путь). Сюда, начиная с Фазы 2 плана, переезжают: Prisma-схема
и клиент БД, авторизация (JWT/сессии), бизнес-логика (сервисы для
journal/work).

## 2026-07-18 — Как /backend стал npm workspace-пакетом, а не просто соседней папкой

**Что нужно сделать.** Разместить `backend/` рядом с `web/` на верхнем
уровне репозитория (явное пожелание — не вкладывать вглубь `web/src/`), но
так, чтобы это оставалось одной Next.js-сборкой/процессом, без отдельного
сервиса.

**Как сделано, шаг за шагом:**

1. Первая попытка — просто путь в `tsconfig.json` (`"@backend/*": ["../backend/src/*"]`),
   без npm-пакета. `next build` (сборщик Turbopack, дефолтный в Next.js 16)
   упал: `Module not found... Import map: aliased to relative
   '../backend/src/_smoke-test' inside of [project]/` — Turbopack
   принципиально не резолвит импорты, ведущие за пределы папки проекта,
   через "голый" путь.
2. Проверено: та же конструкция собирается с флагом `next build --webpack`
   без единой правки — старый бандлер (webpack) такого ограничения не
   имеет. Но это означало бы навсегда отказаться от Turbopack (быстрее,
   дефолт и будущее Next.js) во всех сборках и в `next dev`.
3. Выбран третий путь — **npm workspaces**: `backend/` получил свой
   `package.json` (`name: "@portfolio/backend"`), корень репозитория —
   `package.json` с полем `"workspaces": ["web", "backend"]` (`frontend/`
   сознательно не включён — у него свой независимый цикл установки/сборки,
   трогать его нельfor). `npm install` из корня создаёт symlink
   `node_modules/@portfolio/backend → ../backend`. Теперь это настоящий,
   резолвящийся через `node_modules` пакет — Turbopack прекрасно с этим
   работает, так же, как с любой другой библиотекой.
4. Так как `backend/` не компилируется заранее (нет своего build-шага,
   отдаёт исходный `.ts` прямо из `src/`), в `web/next.config.ts` добавлен
   `transpilePackages: ["@portfolio/backend"]` — иначе Next.js считает
   пакеты внутри `node_modules` уже собранными и не пропускает их через свой
   компилятор.
5. Проверено сборкой (`npm run build` в `web/`, дефолтный Turbopack) —
   `@portfolio/backend` резолвится и попадает в сборку.

**Понятность для другого разработчика.** `backend/package.json` явно
описывает, что это не отдельный сервис (см. его `description`). Импорт в
коде `web/` выглядит как обычная зависимость (`from "@portfolio/backend"`),
а не подозрительный относительный путь наверх (`../../backend/...`), что
понятнее и привычнее.

**Нагрузка.** Никакой лишней нагрузки в рантайме — `transpilePackages`
влияет только на этап сборки, итоговый бандл не отличается от того, как если
бы код лежал прямо в `web/src`.

**Расширяемость/заменяемость.** Самое ценное следствие: если backend
когда-нибудь действительно понадобится вынести в отдельный сервис (см.
план, Фаза 6, "если нужно будет масштабировать backend независимо"),
`@portfolio/backend` уже оформлен как самостоятельный пакет со своими
зависимостями (`backend/package.json`) — не нужно будет "выковыривать" код
из середины Next.js-приложения, только заменить транспорт (прямой вызов
функции → HTTP-клиент).

**Миграция данных / отказоустойчивость.** Не применимо к этому шагу — пакет
пока пустой (`BACKEND_PACKAGE_READY`-заглушка), реальные данные и Prisma
появятся в Фазе 2.

**SOLID.** Dependency Inversion в чистом виде: `web/` зависит от публичного
API пакета `@portfolio/backend` (то, что экспортирует его `src/index.ts`),
а не от внутренней структуры файлов — `backend/` может полностью
переорганизовать свою внутреннюю структуру, не задев `web/`, пока публичный
экспорт из `index.ts` не меняется.

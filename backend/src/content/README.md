# backend/src/content — Document/Block/Post/Work (Фаза 3)

## Зачем сервисный слой, а не прямые Prisma-запросы в страницах

**Будет ли переиспользоваться?** Да, и не один раз: "все посты журнала"
нужны и странице `/journal`, и превью на лендинге ("From the Journal");
"весь список проектов" нужен и `/work`, и блоку "Selected Work" на
лендинге (там только избранные); "проект/пост по slug" нужен и странице
деталей, и перекрёстным ссылкам ("related work"/"related post") с других
страниц. Раз ответ "да" — эти запросы вынесены в `posts.ts`/`work.ts`
именно как переиспользуемые функции, а не написаны заново в каждом
`page.tsx`.

## Унификация Document/Block для постов и кейс-стади

`getDocumentBlocks()` (`document.ts`) — единственное место, которое читает
таблицу `Block`. И `posts.ts` (тело поста), и `work.ts` (нарратив кейс-стади)
вызывают именно её — потому что кейс-стади архитектурно это тот же самый
документ, что и тело поста (решение из фазы проектирования). Если бы это
было два разных запроса в двух разных файлах, любое изменение в том, как
блоки читаются/валидируются (например, добавление пагинации для очень
длинных документов), нужно было бы синхронизировать в двух местах.

## Zod, а не просто TypeScript-интерфейс, для блоков (`blocks.ts`)

`Block.text`/`Block.data` — колонки `Json` в Postgres, то есть на уровне
Prisma это `unknown`, а не типизированные объекты. Zod даёт одно
определение, которое одновременно и валидирует данные в рантайме (на
случай, если в базе окажется что-то неожиданное — опечатка в будущей
админке, ручная правка в базе), и через `z.infer` порождает TypeScript-тип
— вместо того, чтобы поддерживать вручную написанный тип и вручную
написанный валидатор отдельно, которые могут разойтись.

## 2026-07-19 — Переименование `getPublishedPosts` → `getJournalEntries`

**Что нужно сделать.** Функция для списка `/journal`.

**Как сделано.** Первая версия называлась `getPublishedPosts()` и
фильтровала `status: "published"`. Проверка с оригинальной
`JournalListPage` (`frontend/src/pages/journal-list`) показала: страница
рендерит ВСЕ записи, включая `status: "upcoming"` (черновики-анонсы,
просто визуально притушенные) — `getPublishedPosts` с фильтром вернула бы
неполный список, молча потеряв анонсы. Переименовано в
`getJournalEntries()` (без фильтра по статусу, только сортировка по дате),
а фильтр "только опубликованные" вынесен в отдельную
`getLatestPublishedPost()` — то, что действительно нужно только лендингу.

**Понятность для другого разработчика.** Имя функции теперь честно
описывает, что она возвращает — не нужно читать реализацию, чтобы понять,
включены ли черновики.

**Нагрузка/расширяемость/миграция.** Не применимо.

**SOLID.** Interface Segregation: два разных потребителя (страница
`/journal` и превью на лендинге) получили два разных, узких метода вместо
одного метода с булевым флагом-параметром (`getPosts(onlyPublished:
boolean)`), который было бы легче перепутать при вызове.

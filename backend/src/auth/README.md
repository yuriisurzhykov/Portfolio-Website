# backend/src/auth — авторизация (Фаза 2)

Единственный сегодня администратор, но спроектировано так, чтобы несколько
пользователей/устройств и будущий мобильный клиент не требовали редизайна
— см. обсуждение в исходном плане миграции.

## Что здесь и почему именно так

**Access-токен — JWT (`jwt.ts`), refresh-токен — не JWT, а случайная строка
(`tokens.ts`).** Access-токен живёт 15 минут, проверяется по подписи без
обращения к базе (дёшево, можно проверять в `middleware.ts` на каждый
запрос). Refresh-токен живёт 30 дней и обязан быть отзываемым (устройство
потеряли, вышли вручную) — а раз он всё равно всегда сверяется с базой
(`Session.refreshTokenHash`), кодировать его как JWT не даёт вообще ничего,
только увеличивает размер и добавляет второй секрет. Обычная случайная
строка (32 байта, `crypto.randomBytes`) проще и ровно так же безопасна для
этого случая.

**`jose`, не `jsonwebtoken`, для JWT.** `jsonwebtoken` работает только в
Node.js-рантайме (использует модуль `crypto` напрямую). `middleware.ts` в
Next.js по умолчанию выполняется в Edge-рантайме, где модуля `crypto` в
привычном виде нет. `jose` работает поверх Web Crypto API — одинаково в
Node и в Edge, так что не будет сюрприза "middleware не запускается,
потому что библиотека рассчитана только на Node".

**Argon2id, не bcrypt, для паролей (`password.ts`).** Текущая рекомендация
OWASP #1 для хеширования паролей; в `argon2` для Node есть готовые
prebuilt-бинарники под основные платформы, включая Linux/Alpine на VPS —
не проблема сборки из исходников на сервере.

**Refresh-токен хешируется SHA-256 (`tokens.ts`), не Argon2.** Пароль —
что-то, что человек мог выбрать слабым ("123456"), поэтому его хеш должен
быть медленным и дорогим для перебора. Refresh-токен — 256 бит случайных
данных, которые НИКТО не подбирает перебором (перебирать 2^256 вариантов
бессмысленно) — здесь важна быстрая проверка, а не "дорогая" защита от
словарной атаки, которой тут просто нет.

## 2026-07-18 — Смягчение user-enumeration через тайминг при логине

**Что нужно сделать.** `login(email, password)` должен возвращать одинаковую
ошибку и на "email не существует", и на "email существует, пароль неверный"
— иначе злоумышленник может перебором e-mail'ов узнавать, какие аккаунты
существуют.

**Как сделано.** Наивная реализация (`if (!user) return null` до вызова
`verifyPassword`) была бы уязвима: когда пользователя нет, ответ приходит
почти мгновенно (не считая один быстрый SQL-запрос), а когда пользователь
есть — ответ ждёт дорогую argon2-проверку (специально дорогую, это весь
смысл argon2!). Замер времени ответа выдаёт, существует ли email, даже без
знания пароля. Исправлено: если пользователь не найден, код всё равно
прогоняет `verifyPassword` против заранее посчитанного "фиктивного" хеша
(посчитан один раз за время жизни процесса, не на каждый запрос) — время
ответа одинаковое в обоих случаях.

**Понятность для другого разработчика.** Комментарий у `getDummyHash()`
прямо объясняет атаку, которую это предотвращает — без него строка выглядела
бы как непонятный лишний вызов.

**Нагрузка.** Один лишний argon2-вызов на КАЖДЫЙ неудачный логин с
несуществующим email — приемлемо: неудачные попытки логина не горячий путь,
а argon2 всё равно рассчитан на то, чтобы быть в меру дорогим.

**Расширяемость/заменяемость.** Не влияет — чисто внутренняя деталь
`login()`, наружу (HTTP-эндпоинт) виден только `null | AuthResult`.

**Миграция/отказоустойчивость.** Не применимо (нет схемы/данных, только
поведение функции).

**SOLID.** Single Responsibility соблюдён по слоям: `password.ts` не знает
про тайминг-атаки (просто hash/verify), `auth-service.ts` — единственное
место, отвечающее за то, чтобы внешний ответ не утекал информацию о том,
какой из двух факторов (email/пароль) был неверным.

## 2026-07-19 — Тест сразу нашёл реальный баг в `verifyAccessToken`

**Что нужно сделать.** Написать unit-тесты на `jwt.ts` (Vitest,
см. `backend/vitest.config.ts`).

**Как сделано.** Тест `verifyAccessToken("не-jwt-строка")` и
`verifyAccessToken("")` неожиданно не вернули `null`, а бросили необработанное
исключение `JWSInvalid: Invalid Compact JWS`. Причина: `catch`-блок ловил
только два конкретных подкласса ошибок `jose` —
`JWTExpired`/`JWSSignatureVerificationFailed` — а `jose` в реальности бросает
разные подклассы `JOSEError` в зависимости от того, ЧТО не так с токеном:
подпись не совпала — один подкласс; строка вообще не в формате JWT
(мусор/пустая строка) — другой (`JWSInvalid`). Ручное тестирование раньше
проверяло только "испорченный настоящий токен" (подделанная подпись — это
и есть единственный подкласс, который был обработан), поэтому баг не
проявлялся. Исправлено: ловится базовый класс `joseErrors.JOSEError`
целиком — любая ошибка ОТ `jose` означает "невалидный токен", а не
исключительную ситуацию для вызывающего кода.

**Понятность для другого разработчика.** Комментарий в `catch`-блоке прямо
объясняет, почему нужен именно базовый класс, а не перечисление подклассов
— чтобы никто не "исправил" обратно на более узкий список при рефакторинге.

**Нагрузка/расширяемость/миграция.** Не применимо — исправление корректности,
не архитектуры.

**SOLID.** Не впрямую применимо — это просто иллюстрация того, зачем
нужны тесты: баг существовал с самой первой версии файла, прошёл ручное
End-to-End тестирование (Фаза 2) незамеченным, и был найден в первый же
день, когда появились автоматические тесты.

## Ротация refresh-токенов (`session.ts`)

Каждый успешный `refreshSession()` отзывает использованный refresh-токен и
выдаёт новый — токен нельзя использовать повторно. Если украденный
refresh-токен когда-нибудь "срабатывает" ПОСЛЕ того, как легитimный клиент
уже успел обновиться (и старый токен уже отозван), попытка провалится — это
само по себе сигнал компрометации, который стоит логировать в будущем
(вне рамок Фазы 2).

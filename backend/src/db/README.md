# backend/src/db — Prisma Client

## 2026-07-18 — Prisma 7 сломал ожидаемую настройку (`url` в schema.prisma)

**Что нужно сделать.** Подключить Prisma к локальному Postgres (см.
`docker-compose.yml` в корне репозитория) и прогнать первую миграцию для
`User`/`Session`.

**Как сделано.** `npm install prisma` без фиксации версии поставил `7.8.0` —
и `npx prisma migrate dev` сразу упал:
`The datasource property 'url' is no longer supported in schema files`.
Начиная с Prisma 7, ORM стала "Rust-free" (без встроенного бинарника-движка
на Rust) и требует явного driver adapter — подключение к БД больше не
описывается строкой в `schema.prisma`, а настраивается в двух местах:

1. `prisma.config.ts` (в корне пакета `backend/`) — читает `DATABASE_URL`
   для нужд самого Prisma CLI (`migrate`, `studio`, `db push`).
2. `PrismaClient` в коде приложения теперь обязан получать явный
   `adapter` — для PostgreSQL это `@prisma/adapter-pg` поверх обычного `pg`
   (node-postgres) драйвера. Без адаптера конструктор `new PrismaClient()`
   в Prisma 7 бросает ошибку конфигурации.

Не стал закреплять версию на Prisma 6 (старый, привычный способ настройки),
а перешёл на актуальную архитектуру — driver adapters — это теперь
единственный поддерживаемый Prisma путь для новых проектов, а не временный
эксперимент.

**Понятность для другого разработчика.** Комментарий в самом
`client.ts` объясняет ещё одну неочевидную вещь отдельно от Prisma 7 —
зачем нужен singleton через `globalThis` (стандартная и хорошо
задокументированная проблема Next.js hot-reload + Prisma, не специфична для
этого проекта, но легко наткнуться, если не знать).

**Нагрузка.** Driver adapter (`pg` + `@prisma/adapter-pg`) — это чистый
JS/TS без Rust-бинарника Prisma Client прежних версий, что означает меньший
размер зависимостей и более быстрый холодный старт процесса — то, что
особенно ценно на некрупном VPS.

**Расширяемость/заменяемость.** Адаптер отделён от остального кода — весь
проект работает через `prisma` (экспорт из этого файла), а не напрямую с
`pg`/`PrismaPg`; если Prisma сменит рекомендованный адаптер или появится
adapter для другого драйвера, правка нужна только здесь.

**Миграция/отказоустойчивость.** Не применимо к этому шагу (сам клиент, не
данные).

**SOLID.** Dependency Inversion: остальной backend-код зависит от
абстракции `PrismaClient`/`prisma`-экспорта, не от того, что именно `pg`
используется как драйвер под капотом.

## 2026-07-18 — `postinstall: prisma generate`

После `prisma migrate dev` тайпчек (`tsc --noEmit`) неожиданно упал:
`Module '"@prisma/client"' has no exported member 'PrismaClient'`.
Сгенерированный клиент физически не был на месте в тот момент (в npm
workspace он хостится в корневом `node_modules/@prisma/client`, не в
`backend/node_modules`). `npx prisma generate` руками всё исправил, но
полагаться на то, что кто-то не забудет вызвать эту команду руками после
`npm install` — ненадёжно. Добавлен `"postinstall": "prisma generate"` в
`backend/package.json` — теперь генерация клиента гарантированно происходит
при любой установке зависимостей (свежий клон, CI, деплой на VPS), не
только после миграций.

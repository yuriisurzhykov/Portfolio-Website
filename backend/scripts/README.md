# backend/scripts — create-admin

## Почему только интерактивный ввод, без флагов/env для пароля

Пароль администратора не принимается ни как CLI-флаг (остаётся в истории
шелла и в `ps`/Task Manager на время выполнения), ни как переменная
окружения (тогда бы пришлось держать реальный пароль в `.env` файле на
диске). Только интерактивный ввод в консоли — пароль существует в памяти
процесса ровно те несколько секунд, что нужны на хеширование, и нигде не
сохраняется в открытом виде.

## 2026-07-18 — Маскированный ввод пароля зависал при пайпе данных

**Что нужно сделать.** Проверить `create-admin.ts` перед тем, как считать
его готовым.

**Как сделано.** Тест через `echo "email`npass`npass" | npm run create-admin`
завис и был прерван по таймауту. Причина: `promptPassword()` включает
`process.stdin.setRawMode(true)` для маскировки ввода звёздочками — это
работает только с настоящим TTY. У пайпнутого stdin TTY нет, поэтому вызов
либо падает, либо (как здесь) просто никогда не эмиттит нужные события —
скрипт зависает навсегда, без вывода ошибки. Добавлена проверка
`process.stdin.isTTY` — если это не терминал, используется обычный
`readline`-промпт без маскировки (тот же, что для email), вместо
зависания.

**Понятность для другого разработчика.** Без этой проверки поведение
скрипта в CI/автоматизации выглядело бы как необъяснимое зависание —
комментарий в коде прямо называет причину.

**Нагрузка/расширяемость/миграция.** Не применимо.

**SOLID.** Не применимо — защитная проверка окружения выполнения, не
структурное решение.

## 2026-07-18 — Второй, более серьёзный баг: скрипт "успешно" завершался, не создав пользователя (и одна неверная гипотеза по дороге)

**Что нужно сделать.** Проверить сценарий выше ещё раз после фикса с TTY.

**Как сделано, честно включая неверный первый диагноз:**

1. Скрипт отработал с кодом выхода 0, без единой ошибки — но прямой запрос
   к базе (`prisma.user.findMany()`) показал пустой массив: пользователь не
   создался.
2. **Первая гипотеза (оказалась неверной)**: подумал, что дело в том, что
   `prompt()` создавал новый `readline.createInterface` на каждый вопрос и
   закрывал его сразу после ответа — предположил, что при пайпнутом stdin
   это "теряет" уже забуферизованные, но не считанные строки. Переписал на
   один общий `readline.Interface` на весь скрипт.
3. Тест **всё равно упал** — но с другой, куда более информативной
   ошибкой: `Error [ERR_USE_AFTER_CLOSE]: readline was closed`, при попытке
   задать второй вопрос (про пароль). Это вскрыло настоящую причину:
   `readline` в Node.js сам закрывается, когда его input-поток эмиттит
   `'end'` — а при пайпнутом vводе (`echo "..." | script`) весь stdin
   передаётся и сразу закрывается почти одновременно с тем, как
   разрешается первый `question()`. Проблема была не в "потере
   забуферизованных строк между несколькими readline", а в том, что
   **множественные последовательные `question()`-вызовы через readline
   органически ненадёжны на пайпнутом/неинтерактивном stdin** — вне
   зависимости от того, один экземпляр `readline.Interface` или несколько.
   На настоящем интерактивном терминале это никогда не проявляется
   (пользователь физически не закрывает stdin между вопросами).

**Итоговое решение — не бороться с readline+pipe, а изменить архитектуру.**
Вынес саму бизнес-логику (валидация email/пароля, проверка на
существующего пользователя, хеширование, создание записи) в отдельную,
чистую функцию `createAdminUser(email, password)` в
`src/auth/create-admin-user.ts` — её можно вызвать напрямую с
фиксированными аргументами, без stdin/readline вообще. `create-admin.ts`
остался тонкой интерактивной обёрткой вокруг неё, которую нет смысла
пытаться автоматически тестировать через pipe — она проверяется один раз
вручную, как обычный CLI-инструмент для человека.

**Понятность для другого разработчика.** Комментарий в
`create-admin-user.ts` объясняет, почему логика вынесена отдельно —
включая ссылку на то, что порождало проблему, чтобы никто не попытался
"упростить" обратно в один файл и не наткнулся на то же самое.

**Нагрузка.** Не меняется — то же количество операций с БД/хешированием,
просто в отдельной функции.

**Расширяемость/заменяемость.** Заметно лучше: `createAdminUser()` можно
вызвать не только из интерактивного CLI, но и из будущего HTTP-эндпоинта
приглашения нового пользователя (Фаза 4+, если появится мультипользователь)
— бизнес-правило не привязано к тому, что ввод обязательно приходит из
терминала.

**Миграция/отказоустойчивость.** Прямое отношение: без этого фикса
"успешный" (exit code 0) запуск скрипта в каком-то будущем
автоматизированном сценарии (например, provisioning-скрипт при первом
деплое на VPS) мог бы молча не создать администратора — обнаружилось бы
это только при попытке залогиниться, что плохой отказоустойчивый сценарий.
Теперь бизнес-правило само по себе не зависит от того, как в него попали
данные, так что подобный класс "тихого" сбоя для НЕГО в принципе не
существует — можно написать честный unit-тест на `createAdminUser()`
напрямую, без манипуляций со стримами.

**SOLID.** Прямое применение Single Responsibility: "получить учётные
данные от человека" (I/O, тонкая обёртка) и "создать администратора по
готовым email+паролю" (бизнес-правило) — теперь два разных модуля с
разной причиной для изменения. Это же и хороший урок для всего процесса:
"выглядит рабочим при интерактивном запуске" — не то же самое, что
"корректно"; баг был невидим в нормальном интерактивном использовании и
проявлялся только при автоматизированном/пайпнутом vводе, что было ровно
тем сценарием, которым я и тестировал скрипт вместо реального ввода с
клавиатуры.

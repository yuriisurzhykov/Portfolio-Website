# web — Next.js-приложение (SSR + admin + backend)

Это новая версия сайта: Next.js (App Router), рендеринг на сервере, данные из
PostgreSQL, встроенная админка. Заменяет `frontend/` (Vite SPA) после того,
как миграция будет закончена и провалидирована. До этого момента `frontend/`
продолжает деплоиться как обычно — `web/` живёт рядом, не мешая текущему сайту.

Полный план миграции: `.cursor/plans/database-backed_cms_migration_59a172e7.plan.md`
(корень репозитория).

## Журнал решений

### 2026-07-18 — Где живёт новый проект относительно `frontend/`

**Что нужно сделать.** Выбрать расположение нового Next.js-проекта в
репозитории так, чтобы миграция не сломала текущий деплой `frontend/` и
не мешала одновременной разработке.

**Как сделано.** Новый проект создан в `web/` — отдельной директории на
верхнем уровне репозитория, рядом с `frontend/`. Ни один файл `frontend/` не
тронут. `frontend/` продолжает собираться и деплоиться существующим
GitHub Actions workflow (`deploy_release.yaml`) без изменений, пока весь
функционал не будет перенесён и проверен. Переключение (cutover) — отдельный,
более поздний шаг: замена nginx-конфига с раздачи `frontend/dist` на
проксирование в Next.js-процесс, и только после этого `frontend/` можно будет
удалить или архивировать.

**Понятность для другого разработчика.** Разделение прозрачное: два корневых
проекта, у каждого свой `package.json`, свой `README.md`. Не требует
понимания монорепо-инструментов (Turborepo/Nx/pnpm workspaces) — это два
независимых npm-проекта, что соответствует единственному Next.js-приложению
без микросервисной архитектуры backend'а.

**Нагрузка.** Не применимо к этому решению — это вопрос организации кода в
репозитории, а не рантайм-архитектуры.

**Расширяемость/заменяемость.** Полностью: старый `frontend/` можно удалить
одним `git rm -r`, ничего в `web/` от этого не изменится, так как между ними
нет общих файлов и импортов.

**Миграция данных / отказоустойчивость.** Не применимо к этому шагу — данные
появятся начиная с Фазы 2 плана.

**SOLID.** Не применимо к структуре директорий, но по духу — это Dependency
Inversion на уровне инфраструктуры: деплой-пайплайн `frontend/` не зависит от
того, что происходит внутри `web/`, и наоборот.

### 2026-07-18 — Один процесс Next.js, backend — это папка внутри проекта, а не отдельный сервис

> **Уточнено позже в тот же день**: "внутри проекта" в итоге означает не
> `web/src/backend/`, а `/backend` как отдельный npm workspace-пакет рядом с
> `web/` на верхнем уровне репозитория — тот же однопроцессный runtime, но
> более явное разделение по вашей просьбе. Полный разбор, включая находку
> про Turbopack и то, почему обычный путь-алиас не подошёл — в
> `backend/README.md`. Оставляю эту запись ниже как есть — рассуждение о
> "один процесс vs отдельный сервис" не изменилось, изменилось только ГДЕ
> физически лежит папка.

**Что нужно сделать.** Определить, где живёт "backend"-логика (Prisma, работа
с БД, авторизация, бизнес-логика) — в отдельном сервисе (Express/Fastify) или
внутри самого Next.js-приложения.

**Как сделано.** Выбран второй вариант: один Next.js-процесс на VPS, а
backend-логика организована в `src/backend/` внутри этого же проекта.
"Backend" здесь — архитектурный слой (папка с серверным кодом), а не
отдельный деплоймент. Next.js Route Handlers (`src/app/api/**/route.ts`)
и Server Components обращаются к этому слою напрямую, без HTTP-запроса
между "фронтом" и "бэком" — они выполняются в одном процессе.

**Понятность для другого разработчика.** Название папки (`backend`) прямо
говорит о её назначении, что важно, раз мы отказались от буквального
разделения на два сервиса — человек, ожидающий увидеть "papку backend",
её увидит, просто внутри, а не как соседний проект.

**Нагрузка.** Один процесс — меньше сетевых прыжков на каждый запрос
(no HTTP hop между "frontend" и "backend"), что быстрее и не требует
отдельного пула соединений с БД на каждый сервис.

**Расширяемость/заменяемость.** Если в будущем понадобится вынести backend в
отдельный сервис (например, ради независимого масштабирования), код в
`src/backend/` уже изолирован от `src/app/` через чёткую границу импортов —
его можно вынести в отдельный npm-пакет/сервис позже без переписывания
бизнес-логики, только слоя транспорта (Route Handlers → HTTP-клиент).

**Миграция данных / отказоустойчивость.** Не применимо к этому шагу.

**SOLID.** Dependency Inversion: `src/app` (Route Handlers, Server
Components) зависит от абстракций backend-слоя (сервисов), а не от деталей
Prisma/JWT напрямую в каждом файле роута — сами детали спрятаны внутри
`src/backend`.

### 2026-07-18 — Находка: `styled-components` заявлен в зависимостях, но не используется

**Что нужно сделать.** По плану миграции предполагалось настроить SSR registry
для `styled-components` (стандартный шаг при переносе styled-components в
Next.js App Router).

**Как сделано.** Перед тем как тратить время на registry, проверено:
`grep` по всему `frontend/src` на `styled(` и `from "styled-components"` —
ноль совпадений. Пакет висит в `package.json`, но реально не импортируется
ни в одном файле. Вся стилизация — Tailwind v4 + CSS custom properties
(design tokens в `shared/ui/theme/theme.css.ts`, собираются в строку и
вставляются как обычный `<style>`, без runtime styled-components). Значит,
шаг "styled-components SSR registry" из плана — не нужен, и сам пакет в
`web/` не добавляется вовсе.

**Понятность для другого разработчика.** Упрощает картину: один способ
стилизации (Tailwind + CSS-переменные), а не два конкурирующих подхода.

**Нагрузка.** Меньше на один runtime-зависимость (styled-components
генерирует и вставляет `<style>`-теги на клиенте при каждом рендере
компонента) — обычный CSS через Tailwind дешевле по CPU/памяти на слабом
VPS.

**Расширяемость/заменяемость.** Не теряется ничего: если styled-components
понадобится позже для конкретного компонента, его можно добавить точечно, не
меняя остальную архитектуру.

**Миграция данных / отказоустойчивость.** Не применимо.

**SOLID.** Это скорее YAGNI/KISS, чем SOLID напрямую — не тащим
инфраструктуру (SSR registry, доп. babel/next-plugin конфигурацию) под
зависимость, которая нигде не используется.

### 2026-07-18 — Границы Client Component из-за `useTranslation()`/`useTheme()`

**Что нужно сделать.** Next.js App Router по умолчанию рендерит компоненты
на сервере (Server Components) и не даёт им использовать React-хуки вообще
— ни `useState`, ни `useContext`. `useTranslation()` и `useTheme()` из
`shared/i18n`/`shared/theme` — это обёртки над `useContext`, а значит любой
компонент, вызывающий их напрямую, обязан быть Client Component
(директива `"use client"` в первой строке файла).

**Как сделано.** Через `grep` по всему `web/src` на `useTranslation()` /
`useTheme()` / `useState` / `useEffect` / `useContext` найдены все файлы,
которые вызывают хук напрямую (а не только те, что мне казались
интерактивными на глаз) — добавлена директива `"use client"` туда, где её
не было: `Nav`, `AppLayout`, обе detail-страницы, обе list-страницы, секции
лендинга (`Hero`, `Principles`, `TechStack`, `ContactCta`, `SelectedWork`,
`JournalPreview`), `Footer`, оба переключателя в `Nav`, `ContentBlocks`,
`CodeBlock`/`CodeBlockContent`, сам `ThemeProvider`/`I18nProvider` и их
хуки, `ThemeToggle`, `DesignSystemPlayground`, `Storybook`.

**Понятность для другого разработчика.** Директива стоит прямо в файле,
который использует хук — не нужно знать "откуда-то сверху" в дереве
компонентов, что конкретный файл на самом деле клиентский; в этом смысле
явное дублирование директивы в каждом файле-потребителе хука безопаснее,
чем полагаться на неявное наследование границы от родителя.

**Нагрузка.** Существенная часть страницы (весь текст, зависящий от
`pick()`/`ln()`) рендерится на клиенте, а не на сервере — то есть SSR
Фазы 1 даёт SEO/скорость первой отрисовки в первую очередь для разметки и
статических частей (например, `layout.tsx`), но не устраняет клиентский
рендеринг текста целиком. Это ожидаемо и совпадает с решением "не трогать
архитектуру i18n в Фазе 1" — если в будущем понадобится полный SSR для
переведённого текста, i18n-систему придётся пересмотреть отдельно (скорее
всего в сторону чтения словаря на сервере вместо Context на клиенте).

**Расширяемость/заменяемость.** Компоненты, которые НЕ используют
`useTranslation()`/`useTheme()` напрямую (например, чисто презентационные
`Card`, `Tag`, `Text`, `Button`), остались без директивы — они могут в
будущем рендериться и на сервере, если когда-нибудь понадобится страница без
i18n-хуков.

**Миграция/отказоустойчивость.** Не применимо.

**SOLID.** Не затрагивает SOLID напрямую — это ограничение платформы
(Server/Client Component boundary), а не структуры классов/модулей.

### 2026-07-18 — Коллизия имён: FSD-слой `pages/` vs Next.js Pages Router

**Что нужно сделать.** `frontend/` использует feature-sliced-слой с именем
`pages/` (`pages/landing`, `pages/journal-detail`, ...). Это обычное имя
слоя в FSD — но у Next.js есть свой, отдельный от App Router, легаси-роутер
("Pages Router"), который автоматически превращает **любую** папку `pages/`
(в том числе `src/pages/`) в маршруты, по одному файлу на маршрут.

**Как сделано.** При первой сборке (`npm run build`) Next.js упал с ошибкой
`App Router and Pages Router both match path: /storybook` — он
самостоятельно нашёл `src/pages/storybook/` и решил, что это легаси-роут
`/storybook`, что конфликтует с явным `app/storybook/page.tsx`. Никакого
флага "отключить автоопределение Pages Router, но оставить App Router" в
Next.js нет — единственное решение — не называть папку `pages`. Слой
переименован в `views/` (`src/views/landing`, `src/views/journal-detail`,
...), обновлены все 6 импортов из `app/**/page.tsx`.

**Понятность для другого разработчика.** Само по себе `views` вместо
`pages` менее привычно для FSD-документации (там канонично `pages`), так что
это стоит явно знать при онбординге в проект — что здесь `views` это тот же
концепт, что в FSD называется `pages`, просто переименован из-за конфликта
имён с Next.js.

**Нагрузка.** Не применимо.

**Расширяемость/заменяемость.** Не теряется ничего — просто другое имя папки,
весь остальной FSD-слой (widgets/feature/shared/entities) сохраняет
стандартные имена, коллизия была только у этого одного слоя.

**Миграция/отказоустойчивость.** Не применимо.

**SOLID.** Не применимо — вопрос именования, не структуры зависимостей.

### 2026-07-18 — Явные `.ts`/`.tsx` в путях импорта не проходят тайпчек Next.js

**Что нужно сделать.** Часть кода в `frontend/` пишет
`from "./Component.tsx"` (с расширением) — Vite это нормально ест.

**Как сделано.** `next build` компилируется, но падает на этапе тайпчека:
`An import path can only end with a '.tsx' extension when
'allowImportingTsExtensions' is enabled`. Вместо включения этого флага (он
предназначен для специфичных сценариев, не для обычного кода) убраны
расширения `.ts`/`.tsx` из всех путей импорта/экспорта по всему `web/src`
(один regex-проход по всем файлам) — это и есть стандартный, переносимый
способ писать импорты в TypeScript, независимо от Vite/Next.js/webpack.

**Понятность для другого разработчика.** Импорты без расширений — то, что
ожидает увидеть любой TS/React-разработчик; версия с `.tsx` в импорте была
скорее особенностью конкретной настройки Vite, чем осознанным стилем.

**Нагрузка / расширяемость / миграция.** Не применимо — чисто синтаксическая
правка, поведение кода не изменилось.

**SOLID.** Не применимо.

### 2026-07-18 — `lucide-react` закреплён на версии `frontend/`, а не latest

**Что нужно сделать.** Установить те же runtime-зависимости, что использует
`frontend/` (`clsx`, `lucide-react`, `prismjs`, `tailwind-merge`).

**Как сделано.** `npm install lucide-react` без версии поставил `^1.25.0`
(на две мажорных версии новее, чем `^0.553.0` в `frontend/`) — и тайпчек
сразу упал: `Github` больше не экспортируется под этим именем (иконки в
lucide-react периодически переименовываются между мажорными версиями).
Закреплено на точную версию `frontend/` (`lucide-react@0.553.0`) — это не
"более старая/хуже" версия, а гарантия, что набор иконок и их API идентичны
уже проверенному в проде коду, без риска визуальных отличий или новых
поломок API при апгрейде "заодно" с миграцией на Next.js.

**Понятность / нагрузка / расширяемость / миграция / SOLID.** Не
применимо — версия зависимости, не архитектурное решение. Апгрейд
lucide-react до последней версии — отдельная, самостоятельная задача, не
часть миграции на Next.js.

## Структура (целевая, будет заполняться по мере миграции)

```
web/
  prisma/                 # schema.prisma, миграции (стандартное место для Prisma CLI)
  src/
    app/                  # Next.js App Router: только маршруты + Route Handlers, тонкий слой
    backend/              # серверная логика: db-клиент, auth, сервисы (бизнес-логика)
    shared/               # перенесено из frontend/src/shared без изменения логики
    widgets/              # перенесено из frontend/src/widgets
    feature/              # перенесено из frontend/src/feature
    views/                # перенесено из frontend/src/pages — композиция страниц (не роутинг).
                          # Названо "views", не "pages": Next.js резервирует src/pages/ под
                          # свой легаси Pages Router (см. журнал решений выше).
```

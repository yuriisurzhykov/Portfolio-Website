# shared/theme — перенос из frontend/ + найденный hydration-баг (Фаза 3/4 верификация)

## 2026-07-19 — Реальный hydration mismatch на теме, найденный пользователем вживую

**Что нужно сделать.** Проверить сайт после остановки/перезапуска Docker —
пользователь увидел в консоли `Console Error: A tree hydrated but some
attributes of the server rendered HTML didn't match the client properties`
конкретно на кнопках переключателя темы (`aria-pressed`, текст лейбла — не
совпадали между сервером и клиентом).

**Как сделано.** `getInitialPreference()` явно ветвился по
`typeof window === "undefined"` — сервер (где `window` всегда `undefined`)
получал `"dark"`, а клиент (где `window` есть всегда, даже до
гидратации) немедленно читал `localStorage` и мог получить `"light"` или
`"system"`, если пользователь уже сохранял такое предпочтение при
предыдущем визите. Первый рендер клиента и HTML, присланный сервером,
оказывались разными — это ровно тот паттерн, который сообщение об ошибке
React называет по имени как частую причину. Баг существовал в коде с
портирования Фазы 1 (унаследован из `frontend/`), но не мог проявиться
там — в чистом Vite SPA без сервера просто не с чем было расходиться.

Исправлено: `getInitialPreference()` теперь всегда возвращает `"dark"`
(детерминированно, одинаково на сервере и клиенте), а реальное сохранённое
значение читается в `useEffect` (клиент-только, после монтирования) и
применяется через `setPreferenceState`, что естественно запускает уже
существующий эффект синхронизации темы. Тот же принцип уже был в
`resolveTheme()` для случая `"system"` — теперь весь модуль единообразно
никогда не трогает `window`/`localStorage` во время самого рендера, только
внутри эффектов.

**Понятность для другого разработчика.** Комментарий у
`getInitialPreference()` прямо называет паттерн-ловушку по имени
(`typeof window !== "undefined"` в рендере) — так что при похожей правке
где-то ещё этот же баг не повторится незамеченным.

**Нагрузка.** Один лишний `useEffect` на монтирование `ThemeProvider` —
пренебрежимо.

**Расширяемость/заменяемость.** Если позже понадобится ещё одно
клиент-only значение (например, реальные системные метрики), тот же
паттерн — детерминированный дефолт на первый рендер + коррекция в
эффекте — переносится без изменений.

**Миграция/отказоустойчивость.** Побочный эффект фикса: пользователь с
сохранённым `"light"` увидит на долю секунды `"dark"`, прежде чем эффект
скорректирует тему — сознательный, принятый trade-off (тот же класс
проблемы, что и i18n-флеш, см. `shared/i18n/README.md`), предпочтительнее
сломанной гидратации.

**SOLID.** Не применимо напрямую — исправление корректности, не
структуры.

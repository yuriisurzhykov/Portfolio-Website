# shared/ui/icon-picker — IconPickerField / IconRefPreview

## Будет ли переиспользоваться?

Скорее всего да, хотя сегодня есть только один потребитель. Первым
подключён `principles` (карточки "How I work" на лендинге) — но `IconRef`
(схема в `backend/src/content/site-content.ts`) названа не
`PrincipleIcon`, а обобщённо, ровно чтобы `techStack`/любая другая
list-секция могла получить точно такую же пару "тип иконки + значение" в
будущем без переименований. Сам `icon-picker` живёт в `shared/ui`, а не
внутри `views/admin-settings-editor`, по той же причине.

## Зачем вообще эта фича

    10|Пользователь заметил, что у всех "принципов" на сайте нет иконок вообще —
и не потому, что они не отображаются, а потому, что раньше их физически
нельзя было задать: `principlesContentSchema` не имел поля `icon`, а
карточка на лендинге рисовала статичный пустой `<div>`.

## Почему `IconRef` — это `{ type: "none" | "url" | "icon", value?: string }`, а не просто `iconUrl?: string`

Три явных, закрытых состояния, а не один опциональный `iconUrl` +
опциональный `iconName` — иначе обе админ-форма, и `IconRefPreview`
должны были бы сами придумывать правило "что если оба заданы одновременно"
или "что если ни одного". Дискриминированный union в Zod
    20|(`backend/src/content/site-content.ts`) делает это структурно
невозможным — там просто нет состояния, где `value` присутствует, но
неясно, это URL или имя иконки.

`icon: iconRefSchema.default({ type: "none" })` — не `.optional()` — на
уровне схемы: любая уже существующая в базе строка `principles`, у
элементов которой нет ключа `icon` вовсе (весь контент до этой фичи), при
чтении через `getSiteContent` получает `{ type: "none" }` автоматически.
Никакой Prisma-миграции и backfill-скрипта не нужно: `SiteContent.data` —
это один `Json`-столбец (см. `backend/prisma/schema.prisma`), а не
отдельная таблица `Principle` со своими колонками — форма JSON внутри
    30| него — целиком забота Zod-схемы, не SQL.

## Почему валидация иконки — через `lucide-react/dynamic`, а не собственный список из 30-40 имён

Первая версия этого компонента (не попавшая в финальный код) держала
небольшой курируемый `Record<string, LucideIcon>` на ~30-40 подходящих
для "принципов" иконок — с закрытым списком в `<Select>`, где невалидное
значение попросту невозможно ввести. Живой фидбек от пользователя изменил
требование: "хочу видеть live preview при вводе, и чтобы иконка реально
проходила валидацию" — а валидация закрытого списка тривиальна (там просто
нет невалидных значений, нечего проверять). Финальная версия использует
`lucide-react/dynamic`, который для ровно этого сценария существует у
    40| самой библиотеки:

- `iconNames` — настоящий, полный список каждого валидного kebab-case
  имени (`"rocket"`, `"shield-check"`, ...), а не наша копия, которая
  могла бы разойтись с реально установленной версией `lucide-react`.
- `DynamicIcon` — код-сплитящийся рендер по имени (динамический
  `import()` на каждую иконку) — публичный лендинг скачивает только те
  SVG, что реально выбраны в админке, а не весь набор icons.

`lucide-icon-names.ts` — единственное место, знающее про
`lucide-react/dynamic` конкретно (не про `lucide-react` вообще) и про то,
что `iconNames` стоит завернуть в `Set` для O(1)-проверки на каждом
нажатии клавиши, а не гонять `Array.includes` по ~1900 записям на
    50| каждый ре-рендер.

## Почему backend-схема НЕ проверяет `value` против настоящего списка имён

`iconRefSchema`'s `"icon"`-вариант принимает `value: z.string()` без
дополнительной проверки. Проверять на бэкенде против списка
`lucide-react` означало бы вшить снэпшот версии иконок в валидацию
контента — при обновлении `lucide-react` до версии, где иконка
переименована/удалена, старый (уже сохранённый и раньше валидный) контент
начал бы отклоняться на запись без всякой связи с реальным изменением
контента. Вместо этого нераспознанное имя (опечатка, устаревшее имя)
    60| просто не долетает до `DynamicIcon` вообще — `IconRefPreview` заранее
проверяет `isKnownLucideIconName` и рисует тот же нейтральный placeholder,
что и `type: "none"`. Дешёвый, предсказуемый и единообразный fallback,
не стоящий второго источника правды на бэкенде.

## `IconRefPreview` — одна реализация, два потребителя

Публичная карточка (`web/src/views/landing/sections/Principles.tsx`) и
живой preview в самой админ-форме (`IconPickerField`) рендерят иконку
буквально одним и тем же компонентом. Если бы это были два похожих, но
раздельных кастомных рендера — ровно такой сценарий и привёл бы к
    70| исходному багу: preview в админке "работает", а на реальном лендинге
иконка почему-то не показывается (или наоборот).

## Живая проверка

Полностью через настоящий локальный Postgres (в этой песочнице
предварительно поднят через `apt`/`systemd`, никакого Docker не было
под рукой) — не только тесты:

1. `backend/src/content/site-content.test.ts` — round-trip `icon` (оба
   варианта, `url` и `icon`) через реальный `updateSiteContent`/
    80|`getSiteContent`, плюс проверка `.default({ type: "none" })` для строки,
   у которой в JSON нет ключа `icon` вовсе (симулирует контент, записанный
   до этой фичи).
2. Реальный HTTP: залогинен одноразовый admin-пользователь (создан и
   затем удалён после проверки), `PUT /api/admin/settings/principles` с
   `icon: { type: "icon", value: "target" }`, `{ type: "url", value: "..." }`
   и намеренно невалидным именем иконки — все три сохранились и
   корректно прочитались обратно; невалидное имя на публичной странице
   корректно свелось к placeholder, `url`-иконка отрендерилась настоящим
   `<img src="...">` в HTML.
    90|3. Единственное, что curl не может показать — сам `<svg>` для
   `type: "icon"`: `DynamicIcon` резолвит иконку асинхронно (`useEffect` +
   `import()`), так что в исходном SSR HTML её нет, она появляется только
   после гидратации в браузере. Без доступного в этой песочнице браузера
   это проверено `IconRefPreview.test.tsx` — рендер с реальным известным
   именем (`"rocket"`) и `waitFor`, что `<svg>` действительно появляется
   в DOM. Так это и задокументировано — не как "визуально подтверждено",
   а как "механизм резолва подтверждён напрямую", раз браузера не было.

## Тесты

`IconRefPreview.test.tsx` покрывает все состояния: `none`/`undefined` →
   100|placeholder без `<svg>`/`<img>`; `url` → `<img>` с правильным `src`, и
переключение на placeholder после `onError` (битая/недоступная ссылка);
`icon` с настоящим именем → реальный `<svg>` через `DynamicIcon`
(асинхронно, через `waitFor`); `icon` с нераспознанным именем →
placeholder немедленно, без похода в `DynamicIcon` вообще.

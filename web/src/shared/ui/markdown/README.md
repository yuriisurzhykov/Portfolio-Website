# shared/ui/markdown — Markdown

## Будет ли переиспользоваться?

Да — `lead`/`heading`/`paragraph`/`quote`/`note` блоки (`ContentBlocks`) все
рендерят `text` через один и тот же механизм. Раз ответ "да", это отдельный
компонент в `shared/ui`, а не логика, скопированная в каждый `case` свитча
`ContentBlocks`.

## Почему `react-markdown` без `rehype-raw`

Решено на этапе проектирования (см. план миграции): блоки хранят Markdown,
не HTML — это единственный способ поддержать **bold**/*italic*/ссылки без
завода полноценного rich-text редактора в будущей админке (Фаза 4), и без
XSS-поверхности от произвольного HTML в контенте. `react-markdown` без
плагина `rehype-raw` НИКОГДА не интерпретирует HTML-теги в исходном тексте
как разметку — это не побочный эффект, а именно то, ради чего библиотека
выбрана. Подтверждено тестом (`Markdown.test.tsx`) — `<script>`, вставленный
в текст, остаётся видимым, безвредным текстом на странице, не выполняется.

## 2026-07-19 — Добавлен Vitest + React Testing Library в `web/`

**Что нужно сделать.** Проверить, что `<script>` в тексте блока реально не
превращается в живой DOM — это утверждение, а не предположение, и заслуживает
теста, не только визуальной проверки руками.

**Как сделано.** В `web/` до этого не было своего Vitest (только в
`backend/`) — добавлен `vitest.config.ts` (окружение `jsdom` для рендеринга
React-компонентов вне браузера) + `@testing-library/react`. Тест
подключён в `.github/workflows/backend-web-checks.yml` (`npm test` в
`web/`), так что реальный CI прогон, а не только локальный.

**Понятность/нагрузка/расширяемость/миграция/SOLID.** Не применимо
отдельно — это тестовая инфраструктура, не архитектурное решение
продакшен-кода.

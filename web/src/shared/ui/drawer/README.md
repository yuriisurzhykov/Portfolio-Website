# shared/ui/drawer — Drawer

## Будет ли переиспользоваться?

Да — созданный для мобильного меню (`widgets/nav`), но спроектирован как
универсальный слайд-панель-компонент, а не логика внутри `Nav.tsx`.
Следующий очевидный потребитель — мобильная навигация будущей админки
(Фаза 4): та же механика (подложка, закрытие по Escape/клику вне,
блокировка скролла страницы) нужна там один-в-один, различается только
содержимое панели.

## Доступность

- `role="dialog"` + `aria-modal="true"` — стандартная семантика модального
  диалога.
- `aria-hidden`/`inert` синхронизированы с `open` — когда панель визуально
  скрыта (задвинута за экран через `translate-x`), она одновременно
  выпадает из accessibility-дерева И из tab-порядка. `aria-hidden` одного
  без `inert` недостаточно — читалки экрана его уважают, а вот
  клавиатурный `Tab` без `inert` всё равно может увести фокус на скрытую,
  но технически ещё присутствующую в DOM кнопку.
- Блокировка скролла `<body>` пока панель открыта, восстановление
  предыдущего значения (не жёсткий `""`) при закрытии/размонтировании —
  на случай, если что-то ещё до этого уже трогало `overflow`.
- Escape закрывает панель; закрытие по клику на подложку — стандартное
  поведение любого модального оверлея.

## 2026-07-19 — Найдено на живом скриншоте: `backdrop-filter` предка ломает `position: fixed`

**Что нужно сделать.** Пользователь открыл мобильное меню и увидел не
полноэкранную панель, а нечто "сжатое" в узкую полоску сверху экрана, с
остальным контентом, просвечивающим сквозь.

**Как сделано.** Причина — не z-index (первая гипотеза пользователя), а
менее очевидное правило CSS: `backdrop-filter` (как и `filter`,
`transform`, `perspective`, `will-change` с этими свойствами) на
ПРЕДКЕ создаёт новый containing block для потомков с `position: fixed`.
`<Drawer/>` рендерился внутри `<header>` у `Nav`, а у самого `header` есть
класс `backdrop-blur-md` (`backdrop-filter: blur(...)`, полупрозрачная
подложка шапки) — поэтому "fixed" подложка и панель `Drawer` фиксировались
не относительно экрана, а относительно самого `header`, чья высота — одна
строка. Всё "полноэкранное" визуально сжалось в размер шапки.

Точечный фикс (вынести `Drawer` за пределы `<header>` в `Nav.tsx`) решил
бы именно этот случай, но оставил бы точно такую же мину для следующего
места, где `<Drawer/>` окажется внутри чего-то с `filter`/`transform` —
ровно то, что не должно быть заботой каждого вызывающего кода для общего,
переиспользуемого компонента. Вместо этого `Drawer` теперь рендерится через
`createPortal` прямо в `document.body` — независимо от того, где в React-
дереве он вызван, его DOM-узел всегда физически лежит вне любых предков и
их CSS.

**Понятность для другого разработчика.** Подробный комментарий в
`Drawer.tsx` объясняет само правило CSS по имени (containing block для
`position: fixed`) — без этого баг выглядит необъяснимым волшебством,
"почему высота такая маленькая", если не знать про `backdrop-filter`.

**Нагрузка.** Портал — это просто другое DOM-размещение того же дерева
React-компонентов, не дублирование рендера.

**Расширяемость/заменяемость.** Главный смысл фикса: теперь `Drawer`
можно безопасно вызвать из ЛЮБОГО места, включая внутри чего угодно со
`filter`/`transform`/`backdrop-filter` — компонент не полагается на то, что
вызывающий код "будет aккуратен".

**Миграция/отказоустойчивость.** Не применимо.

**SOLID.** Liskov-подобное рассуждение: `Drawer` должен вести себя
одинаково корректно независимо от того, ГДЕ в дереве его использовали —
раньше это было не так (поведение "по-умолчанию ломается внутри
`backdrop-filter`"), теперь это инвариант, который компонент сам
гарантирует, а не оставляет на совесть каждого места использования.

## Тесты

`Drawer.test.tsx` покрывает: `aria-hidden`/`inert` синхронизацию с `open`,
закрытие по клику на подложку, закрытие по Escape (и что Escape НЕ
закрывает то, что уже закрыто — проверка, что обработчик не навешен
лишний раз), блокировку/восстановление скролла `<body>`.

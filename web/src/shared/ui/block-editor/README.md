# shared/ui/block-editor — DraftBlock + BlockListEditor

## Будет ли переиспользоваться?

Да, и это буквально причина, почему этот компонент существует как один
общий слайс, а не два похожих редактора: `admin-post-editor` использует
его для тела поста, `admin-work-editor` — для нарратива кейс-стади. Оба
редактируют "список блоков документа" — то же архитектурное решение
(кейс-стади это тот же `Document`, что и тело поста), что уже применялось
на read-стороне (`shared/ui/content-blocks`) и на backend-стороне
(`backend/src/content/document.ts`), теперь применено на write-стороне.

## Зачем `DraftBlock`, а не редактирование `Block`/`BlockInput` напрямую

**Что нужно сделать.** Редактор должен позволять добавлять/удалять/
переставлять блоки восьми разных форм (Lead/Heading/Paragraph/Quote/Note/
Image/Code/ApproachList) и редактировать поля каждого без падений на
`undefined`.

**Как сделано.** `DraftBlock` (`draft-block.ts`) — собственная форма
блока для формы, НЕ то же самое, что `Block`/`BlockInput` из
`@portfolio/backend`. Два отличия, оба — ровно то, что нужно форме, а не
хранилищу:

1. `clientId` вместо `id`/`order` — React key при живом
   добавлении/удалении/перестановке в браузере; никогда не отправляется на
   сервер (порядок — позиция в массиве на момент сохранения, id назначает
   база — см. `replaceDocumentContent` в backend).
2. Каждое поле - плоское, всегда-строка/всегда-объект верхнего уровня
   (`level: 2 | 3` вместо `data?.level`, `width: string` вместо
   `data.width: number | undefined`) — чтобы `<input value=... onChange=...>`
   в `BlockFields.tsx` не делал null-проверку `data` на каждое нажатие
   клавиши. Конвертация в/из формы `Block`/`BlockInput` происходит только
   на границе (`blockToDraft`/`draftToBlockInput`), не размазана по полям.

Переупорядочивание — кнопки "вверх"/"вниз", не drag-and-drop: полностью
доступно с клавиатуры и скринридера без какой-либо дополнительной работы,
и не тянет DnD-библиотеку ради обычно 3-10 блоков на документ — тот же
принцип "простая админка", что и во всём плане.

**Понятность для другого разработчика.** Комментарий на `DraftBlock`
явно перечисляет оба отличия от `Block` и объясняет, зачем каждое —
без него неочевидно, почему это не просто "тот же тип с добавленным
`clientId`".

**Нагрузка.** Не применимо — форма из нескольких блоков, не список на
тысячи элементов.

**Расширяемость/заменяемость.** Добавление нового типа блока — три места,
не больше: `blocks.ts` (backend, одно "core"-определение), `draft-block.ts`
(добавить вариант в `DraftBlock` + ветку в трёх функциях конвертации +
запись в `BLOCK_TYPE_OPTIONS`), `BlockFields.tsx` (ветка в `switch`).
`BlockListEditor.tsx` не меняется вообще — он работает с `DraftBlock`
целиком, не знает про конкретные типы.

**Миграция/отказоустойчивость.** Не применимо (клиентское состояние формы,
не персистентные данные).

**SOLID.** Interface Segregation: `BlockFields` получает `draft` и
`onChange`, ничего не знает про то, что это блок №3 из 7 в списке —
`BlockListEditor` полностью владеет add/remove/reorder, `BlockFields`
полностью владеет полями одного блока. Open/Closed на уровне
`BlockListEditor`: добавление нового типа блока не требует трогать этот
файл, только `BlockFields`/`draft-block.ts`.

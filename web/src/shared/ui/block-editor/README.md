# shared/ui/block-editor — BlockNote WYSIWYG editor (replaces `BlockListEditor`/`BlockFields`)

## Будет ли переиспользоваться?

Да, ровно как и раньше: `admin-post-editor` использует `<BlockEditor>` для
тела поста, `admin-work-editor` — для нарратива кейс-стади, а с этой
фазой к ним добавились `admin-post-translate`/`admin-work-translate` —
русский перевод тела/кейс-стади это тот же самый `<BlockEditor>`, просто с
другим начальным содержимым (см. `backend/src/content/README.md`'s
unification rationale — тело поста и кейс-стади это один и тот же
`Document`, теперь ещё и русский перевод — тоже просто `Document`, не
отдельная форма).

## Зачем полностью новый компонент, а не доработка `BlockListEditor`

**Что нужно сделать.** Заменить форму "выбери тип блока из списка → нажми
Add block → заполни поля" на настоящий WYSIWYG: слэш-меню для вставки
блока, тулбар форматирования (**bold**/*italic*/ссылки) при выделении
текста, смену типа блока через тот же слэш-интерфейс.

**Как сделано.** [BlockNote](https://www.blocknotejs.org/) (`@blocknote/core`
+ `@blocknote/react` + `@blocknote/mantine`) — редактор поверх
ProseMirror/Tiptap с собственной схемой блоков, слэш-меню и тулбаром "из
коробки". Старый `BlockListEditor`/`BlockFields`/`draft-block.ts`
(`DraftBlock`, add/remove/reorder кнопки, восемь отдельных мини-форм)
удалены целиком, а не доработаны — они решали ровно проблему "форма для
редактирования полей", которую WYSIWYG-редактор не имеет вообще (нет ни
`DraftBlock`, ни кнопок "вверх"/"вниз": переставить блок в BlockNote — это
просто перетащить его).

### Схема (`schema.ts`) — почему не "все блоки — кастомные" и не "все — дефолтные"

`BlockNoteSchema.create({ blockSpecs: {...} })` с явным списком, не
`.create().extend({...})` (дефолты + добавки) — сайт рендерит ровно 8 типов
блоков (`backend/src/content/blocks.ts`), а полный дефолтный набор
BlockNote (списки, таблицы, video/audio/file, свой `codeBlock`) дал бы
редактору создавать контент, для которого у `<ContentBlocks>` просто нет
кейса.

Только `paragraph` и `heading` — настоящие дефолты BlockNote (`heading`
переконфигурирован на уровни 2/3 — `createHeadingBlockSpec({ levels: [2, 3] })`
— у поста/кейс-стади нет визуального `<h1>`, это уже `<h1>` страницы).
Все остальные — кастомные (`blocks/*.tsx`), и не потому что "все кастомные
интереснее", а по конкретной причине на тип:

- **`quote`** — переопределён, не расширен: у встроенного `quote` нет
  `attribution`, только `backgroundColor`/`textColor`.
- **`note`** — нет дефолтного аналога вообще (callout с вариантом
  info/warning/tip).
- **`image`** — свой `alt` (обязательный, для доступности) вместо
  дефолтных `name`/upload-виджета, которому тут не с чем работать (нет
  backend-эндпоинта загрузки — как и раньше, картинки лежат в
  `web/public/`, это просто текстовое поле URL).
- **`code`** — свой `filename`, которого у встроенного `codeBlock` нет;
  сам текст кода — `props.code` (обычный `<Textarea>`), не BlockNote
  rich-text — код не должен получать bold/italic из тулбара.
- **`approachList`** — нет дефолтного аналога вообще; список
  title+description пар нет способа выразить как BlockNote-проп напрямую
  (`PropSchema` — только `boolean | number | string`), поэтому это один
  JSON-строковый проп `itemsJson` (`parseApproachItems`/`JSON.stringify`)
  с собственным add/remove/edit UI внутри блока — тот же UX, что был у
  старого `ApproachListFields`, просто как нода внутри канвы редактора, а
  не отдельная секция формы.

### Конвертация (`convert.ts`) — почему через "голый paragraph", а не напрямую

**Что нужно сделать.** Хранилище (Postgres) видит блок как plain-string
`text`/`data` (`backend/src/content/blocks.ts`) — редактор видит его как
BlockNote-блок с rich-text `content` (список inline-нод со стилями). Нужна
граница между двумя формами в обе стороны, причём **bold**/*italic*/ссылки
должны реально сохраняться как Markdown, не потеряться.

**Как сделано, и какой был неверный первый заход.** Первая идея — звать
`editor.blocksToMarkdownLossy([block])` прямо на блоке "heading"/"quote"/
"note" целиком. Не подошло: эти методы сериализуют блок ПОЛНОСТЬЮ, включая
его собственный блочный Markdown-синтаксис — `"## "` для heading, `"> "`
для quote — а `type` блока в этой схеме уже кодирует "это heading", хранить
`"## "` ещё и внутри `text` означало бы задвоение (на публичной странице
`<ContentBlocks>` уже сама решает, что это `<h2>`). Исправлено: перед
сериализацией/парсингом content оборачивается в "голый" `{ type:
"paragraph", content }` — у paragraph нет собственного префикса, так что
результат — ровно inline-разметка, ничего больше. Симметрично в обе
стороны: `markdownToInlineContent`/`inlineContentToMarkdown`.

Парсинг (`tryParseMarkdownToBlocks`) — метод РЕДАКТОРА, не отдельная
утилита (Markdown-поддержка в BlockNote завязана на ту же
ProseMirror-схему, что и сам редактор), но оба метода в этой версии
BlockNote синхронные (не `Promise` — проверено по `ExportManager.d.ts`),
поэтому `blocksToPartialBlocks` создаёт свой одноразовый, никогда не
монтируемый `BlockNoteEditor.create({ schema })` только для парсинга —
`.create()` без последующего `.mount(element)` не создаёт реальный
ProseMirror `EditorView`, так что ничего не нужно чистить/уничтожать после
использования.

### `BlockEditor` — почему `ref.getBlocks()`, а не `value`/`onChange`

Старый `BlockListEditor` был полностью controlled: `blocks`/`onChange`,
родитель (`PostEditorPage`) хранил `DraftBlock[]` в своём `useState` и
пере-рендерился на каждое изменение. `<BlockEditor>` — нет: BlockNote
(`useCreateBlockNote`) уже держит живой ProseMirror-документ всё время,
пока редактор смонтирован; зеркалить каждую нажатую клавишу обратно в
родительский `Block[]`/`BlockInput[]` (гоняя `blocksToMarkdownLossy` на
каждый символ) — чистая трата, если это значение не нужно никому до
момента отправки формы. Вместо этого `BlockEditor` — `forwardRef` с
`useImperativeHandle`, отдающий `getBlocks(): BlockInput[]` — вызывается
ровно один раз, в `handleSubmit` вызывающей страницы.

## 2026-07-19 — Два реальных краша, найденных вживую после первого деплоя (не в тестах)

**Что произошло.** Пользователь открыл `/admin/journal/[slug]/edit` для
поста с блоком кода и получил `Runtime Error`. Оба нижеописанных бага —
классический пример того, почему "вручную проверено в браузере" не
опционально: юнит-тесты (jsdom) прошли зелёными, `next build` прошёл, но
jsdom и настоящий Next.js SSR — не одна и та же среда.

**Баг 1 — `"code"` не может быть одновременно блоком и inline-стилем.**
Кастомный блок `code` был зарегистрирован в BlockNote под тем же именем
`"code"`, что и дефолтный inline-стиль `code` (форматирование `` `код` ``
в тексте, часть `defaultStyleSpecs`, который эта схема не переопределяла).
ProseMirror требует, чтобы одно имя в схеме было либо node, либо mark,
никогда оба — конструктор редактора падал с `RangeError: code can not be
both a node and a mark`, что BlockNote оборачивает в куда менее понятное
"Error creating document from blocks passed as `initialContent`" (и это
сообщение появлялось даже для ЧУЖОГО, не связанного с кодом временного
редактора внутри `blocksToPartialBlocks` — оба редактора используют одну
и ту же схему, так что ошибка конструктора схемы бьёт по обоим).
Диагностировано не по описанию ошибки (оно указывало на неправильную
строку), а прогоном `BlockNoteEditor.create({schema})` изолированно в
vitest и чтением `error.cause` — реальная причина была в `cause`, не в
верхнем сообщении. Исправлено переименованием BlockNote-внутреннего типа
в `"codeSnippet"` (`blocks/CodeBlock.tsx`) — формат хранения
(`BlockInput.type: "code"`) не тронут, конвертация в `convert.ts` просто
маппит одно имя на другое на границе.

**Баг 2 — BlockNote не может выполняться при серверном рендере.** Даже
после фикса №1 страница редактирования падала уже по-другому:
`ReferenceError: document is not defined`. Next.js рендерит Client
Component (директива `"use client"` тут не означает "никогда не
выполняется на сервере" — означает "может выполняться на клиенте")
СНАЧАЛА на сервере для первого HTML, и только потом гидрирует на клиенте
— `useCreateBlockNote`/`BlockNoteEditor.create()` полагаются на реальный
браузерный DOM (`document`), которого в Node.js-процессе Next.js просто
нет. Это задокументированное собственное ограничение BlockNote (см. их
гайд "With Next.js" — "make sure Next.js does not try to render
BlockNote as a server-side component"), не что-то специфичное для схемы
этого проекта. Официальная рекомендация — `next/dynamic(..., { ssr:
false })` вокруг всего компонента редактора; здесь применён
функционально эквивалентный "mounted-гейт": `BlockEditor` (тонкая
обёртка) рендерит placeholder до тех пор, пока `useEffect` (который
никогда не выполняется при SSR) не переключит `mounted` в `true`, и
только тогда монтирует `MountedBlockEditor` — весь код, реально
касающийся BlockNote, недостижим до этого момента вообще, а не просто
условно вызывается. Выбран этот вариант, а не `next/dynamic`, чтобы
избежать вопроса, как `next/dynamic` прокидывает `ref` через свой
lazy-loading wrapper — здесь `ref` идёт прямо в настоящий
`useImperativeHandle` внутри `MountedBlockEditor`, без прокси-слоя.

**Почему юнит-тесты этого не поймали.** `convert.test.ts` (ниже) вызывает
`BlockNoteEditor.create()` напрямую в vitest — тестовое окружение jsdom
ПРЕДОСТАВЛЯЕТ `document` (это буквально его смысл), так что баг 2 в
принципе не мог проявиться там. Баг 1 тест бы поймал (и теперь ловит,
`convert.test.ts` использует настоящий конструктор редактора, не только
проверяет форму данных) — но только тогда, когда сам тест существовал;
до этой сессии его не было — конвертация проверялась только на форме
JSON, не на факте, что схема с этими данными реально собирается.

**Понятность для другого разработчика.** Комментарий в `BlockNoteEditor.tsx`
прямо объясняет, почему `BlockEditor` — не один компонент, вызывающий все
хуки условно (это сломало бы правило "одинаковый порядок хуков на каждый
рендер"), а два компонента, один из которых просто не существует до
маунта.

**SOLID.** Не пострадал структурно — `MountedBlockEditor` содержит ровно
тот же код, что раньше был во всём `BlockEditor`; единственное изменение
— условие, ПРИ КОТОРОМ этот код вообще выполняется.

## Тесты и проверка

Юнит-тестов на сам смонтированный `<BlockEditor>` (клики по тулбару,
слэш-меню как UI) нет — BlockNote внутри использует реальный
ProseMirror/Tiptap `EditorView`, которому для такого теста нужна
настоящая DOM-среда с contentEditable-семантикой, которую jsdom не
реализует достаточно полно (типичное ограничение для rich-text
редакторов; см. `@testing-library`'s собственные предупреждения про
contentEditable). Но `convert.test.ts` ДЕЙСТВИТЕЛЬНО конструирует
настоящий `BlockNoteEditor.create({schema, initialContent})` (не только
проверяет форму JSON, которую возвращают `blocksToPartialBlocks`/
`editorBlocksToBlockInputs`) — именно этот тест поймал бы Баг 1 выше
(`"code"` как node и mark одновременно), если бы существовал раньше;
теперь он часть обычного прогона `vitest run`, а не только ручной
проверки. Баг 2 (SSR) jsdom не может поймать в принципе (jsdom
предоставляет `document`, а сама суть бага — его отсутствие в Node.js
без jsdom) — здесь единственная реальная проверка это ручной прогон
через `next build && next start` и настоящий HTTP-запрос к
`/admin/journal/[slug]/edit`/`/admin/work/[slug]/edit` для контента,
у которого есть код-блок (см. оба журнала выше). Дополнительно проверено
вручную: создание/редактирование каждого из 8 типов блоков, слэш-меню,
тулбар форматирования, сохранение и перезагрузка страницы (round-trip
через backend) — см. корневой план, "Тесты и проверка".

## SOLID

Single Responsibility: `schema.ts` знает только "какие блоки существуют и
как они выглядят", `convert.ts` — только "как переводить между
хранилищем и редактором", `slash-menu.ts` — только "что показывать по
`/`", `BlockNoteEditor.tsx` — только "смонтировать редактор и отдать
`getBlocks()`". Open/Closed: новый тип блока — это новый файл в `blocks/`
плюс одна строка в `schema.ts`'s `blockSpecs` и один пункт в
`slash-menu.ts` — ни `convert.ts`, ни `BlockNoteEditor.tsx` не меняются
структурно (только добавляется `case` в двух `switch`).

# shared/ui/admin-list-item — AdminListItem

## Будет ли переиспользоваться?

Да, с первого дня: `AdminJournalListPage` и `AdminWorkListPage` рендерили
буквально один и тот же `<Card>`-макет (title+badge слева, slug+meta
строкой под ним, Edit/Delete справа) — два места, значит выносится сразу,
а не откладывается до третьего (см. `development-methodology.mdc`'s
"Будет ли переиспользоваться?").

## 2026-07-20 — Живая обратная связь по скриншоту: строка списка ломалась на узких экранах

**Что нужно сделать.** Пользователь прислал скриншот: на узкой панели
title/slug/дата/бэйдж/кнопки — всё в одну строку через
`justify-between` — не помещалось, slug переносился посимвольно, кнопки
наезжали на текст. Запрошена конкретная структура: статус слева-сверху,
дата справа-сверху, заголовок под ними (можно в несколько строк, но не
больше трёх), slug под заголовком (с оговоркой "нужен ли вообще"), и
Edit/Delete — на одной строке с заголовком, справа.

**Как сделано.** Вертикальный `flex-col` с тремя явными строками вместо
одной строки, тянущейся горизонтально: (1) бэйджи слева + meta справа,
(2) заголовок (`line-clamp-3` — обрезает многострочный текст троеточием
после трёх строк, высота карточки при этом свободно следует за реальной
длиной заголовка, а не фиксирована) и кнопки, выровненные по верху
(`items-start`, не `items-center` — чтобы кнопки не "плавали" по центру
блока заголовка, если он растягивается на три строки), (3) slug —
опционален (`slug?: string`, строка рендерится только если он передан) —
раз пользователь сам не уверен, нужен ли он, компонент не заставляет
выбор навсегда, следующий шаг ("на самом деле убрать") — правка вызова,
не компонента.

**"Иконка становится текстом по анимации" — как это на самом деле сделано.**
Не растущий текст РЯДОМ с иконкой (что было бы проще, но не совпадает с
формулировкой "становится") — иконка и текст лежат друг НАД другом
(оба `absolute inset-0`, оба центрированы) в одной и той же квадратной
кнопке, и на hover/focus один просто гаснет (`opacity` → 0), а другой
проявляется (`opacity` → 100) — настоящий кросс-фейд, не два элемента
рядом. Сама кнопка при этом расширяется (`w-9` → `w-[4.5rem]`), чтобы
слово "Edit"/"Delete" не обрезалось — то есть контрол одновременно и
"морфит" содержимое, и раздвигается под него, но НЕ сдвигает соседние
элементы (кнопка `shrink-0`, растёт только сама кнопка, не строка).

**`tone="danger"` у Delete — красный только на hover, не всегда.**
Постоянно красная кнопка Delete в списке из нескольких карточек читается
как "тут что-то не так" с первого взгляда, даже когда всё в порядке;
красный должен появляться в момент, когда пользователь СОБИРАЕТСЯ
удалить, не постоянно.

**Понятность для другого разработчика.** `RevealAction`'s пропсы —
дискриминированный union по `as: "link" | "button"` (`Link` для Edit,
ведёт по `href`; `<button onClick>` для Delete, у которого есть
`loading`, а у навигации по ссылке нет) — TypeScript не даст передать
`loading` в вариант `"link"`, что документирует "у Edit нет состояния
загрузки" на уровне типов, не только в комментарии.

**Нагрузка.** Не применимо — статичная разметка + CSS-переходы, ни одного
JS-таймера/слушателя сверх обычных `onClick`/`onDelete`.

**Расширяемость/заменяемость.** Третий потребитель (если появится) не
требует трогать `AdminListItem` вообще — `badges`/`meta` это `ReactNode`,
не типизированный `status`, так что любая будущая сущность со своим
набором бэйджей и своим форматом meta-строки подходит без изменений.

**Миграция/отказоустойчивость.** Не применимо.

**SOLID.** Single Responsibility: `AdminListItem` не знает про `Post`/
`Work` вообще (только про badges/meta/title/slug/editHref/onDelete) —
`AdminJournalListPage`/`AdminWorkListPage` решают, ЧТО подставить в каждый
слот, `AdminListItem` только решает, КАК это расположить и анимировать.

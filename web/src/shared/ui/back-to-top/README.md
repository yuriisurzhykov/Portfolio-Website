# shared/ui/back-to-top — BackToTop

## Будет ли переиспользоваться?

Да, в том же смысле, что и `Drawer`: это не "часть шапки", а самостоятельный
плавающий элемент интерфейса всего сайта, подключаемый один раз в
`AppLayout` рядом с `<Nav/>`/`<Footer/>`, а не внутри `Nav.tsx`. Если позже
появится страница/раздел без общего `AppLayout` (например, отдельный layout
будущей админки, Фаза 4), кнопку можно подключить туда тем же способом —
компонент не завязан на разметку шапки.

## 2026-07-19 — Header скрывается при скролле вниз, появляется при скролле вверх + FAB «наверх»

**Что нужно сделать.** Пользователь: шапка (`sticky`) должна вести себя
как sticky-заголовок с авто-скрытием — видна у hero-секции, прячется при
скролле вниз, возвращается при скролле вверх. Плюс отдельная просьба
посоветовать: в момент, когда шапки не видно, показывать плавающую кнопку
«наверх страницы».

**Как сделано.** Общее условие "видимость шапки" вынесено в
`shared/lib/useHideOnScroll.ts` — единственный переиспользуемый хук в этом
файле (в отличие от соседних `useUrlHash`/`useScrollToLocation`, которые
остаются локальными по YAGNI до второго места использования — см.
`widgets/nav/README.md`), потому что здесь он используется РОВНО в двух
местах с инвертированным условием: `Nav` скрывает себя, когда хук вернул
`true`, `BackToTop` показывает себя тем же самым `true`. Оба компонента
вызывают хук независимо (два отдельных `scroll`-listener'а, а не один
подъём состояния в `AppLayout`) — осознанный компромисс простоты для сайта
такого масштаба, см. комментарий в самом `BackToTop.tsx`.

Логика хука: `revealThresholdPx` (по умолчанию 120px) — пока прокрутка
меньше этого порога, шапка всегда видна (это и есть "sticky на
hero-секции"); выше порога решение зависит от направления последнего
скролла. Обработчик throttled через `requestAnimationFrame`, а не вызывает
`setState` на каждое событие `scroll` — иначе одно движение колёсиком
мыши даёт десятки лишних ре-рендеров в кадр.

`BackToTop` — обычная `fixed`-кнопка с иконкой `ArrowUp`, скроллит
`window.scrollTo({top: 0, behavior: "smooth"})`. Скрытое состояние — не
только `opacity-0`, но и `aria-hidden` + `tabIndex={-1}`, тем же
рассуждением, что и `inert` у `Drawer` (см. его README): невидимый элемент
не должен быть доступен по Tab.

**Понятность для другого разработчика.** Комментарий в `useHideOnScroll.ts`
явно объясняет, ПОЧЕМУ он вынесен в `shared/lib`, а не остался локальным —
без этого объяснения неочевидно, почему для этого хука сделано исключение
из правила "не абстрагировать, пока не появится третье место".

**Нагрузка.** Два независимых `scroll`-listener'а (`Nav` + `BackToTop`),
каждый throttled через `requestAnimationFrame` — на практике не более
одного `setState` на компонент за кадр, независимо от частоты событий
`scroll`. Для сайта такого масштаба это не заметно; если появится третий
потребитель этого же условия, стоит пересмотреть в сторону одного
поднятого состояния (см. комментарий в `BackToTop.tsx`).

**Расширяемость/заменяемость.** `revealThresholdPx` — параметр хука, а не
константа внутри тела — оба текущих вызова используют дефолт, но ничто не
мешает `BackToTop` использовать другой порог, если понадобится.

**Миграция/отказоустойчивость.** Не применимо (нет данных).

**SOLID.** Single Responsibility: `useHideOnScroll` знает только про
"скроллили вниз/вверх относительно порога", ничего не знает ни про шапку,
ни про кнопку — оба компонента сами решают, что делать с булевым
результатом (в противоположные стороны).

## Тесты и находка про `vi.useFakeTimers()` + `requestAnimationFrame`

`useHideOnScroll.test.ts` — 5 тестов на само условие скрытия (старт видим,
остаётся видим в пределах порога, скрывается за порогом, снова видим при
скролле вверх, снова видим у самого верха независимо от направления).

Находка при написании тестов, потенциально полезная для любого будущего
теста, троттлящего `scroll`/`resize` через `requestAnimationFrame`:

1. `vi.useFakeTimers()` сам по умолчанию подменяет `requestAnimationFrame`/
   `cancelAnimationFrame` (это одни из имён в его списке `methods`) — ручной
   `vi.stubGlobal("requestAnimationFrame", ...)`, вызванный ПОСЛЕ
   `vi.useFakeTimers()` (или до — порядок не важен, `useFakeTimers()`
   в любом случае переустанавливает их своими), просто перезатирается и
   никогда не вызывается. Правильный способ — не подменять
   `requestAnimationFrame` вручную вообще, а довериться встроенной фейковой
   реализации и продвигать её через `vi.advanceTimersToNextFrame()`.
2. Даже с `vi.advanceTimersToNextFrame()` результат не отражался в
   `result.current` от `renderHook`, пока продвижение таймера не было
   обёрнуто в `act()` из `@testing-library/react`. Без `act()` React
   технически обновляет состояние, но тестовая обёртка не гарантирует
   синхронный флаш перед следующим чтением `result.current` — маскируется
   под "хук не работает", хотя реальная причина — тест не подождал React.

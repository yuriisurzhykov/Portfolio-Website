# shared/i18n — перенос из frontend/ (без изменения дизайна, с одним фиксом)

Перенесено из `frontend/src/shared/i18n` практически без изменений: два
независимых механизма продолжают жить рядом, как и раньше —
`pick(localizedValue)` для двуязычных объектов прямо в данных (`{en, ru}`,
не требует загрузки словаря) и `ln("key")` для словарных строк UI-хрома
(кнопки, лейблы навигации), которые подгружаются из
`/locales/{en,ru}/translation.json` через `fetch()`.

## 2026-07-18 — Найденный краш при переносе на Next.js

**Что нужно сделать.** Перенести `I18nProvider` так, чтобы `ln()` не падал.

**Как сделано.** В старом Vite-приложении `main.tsx` делал `await initI18n()`
на уровне модуля **до** первого `ReactDOM.render(...)` — то есть к моменту,
когда `I18nProvider` монтировался, оба словаря уже были загружены, и
`setLocale('en')` внутри его `useEffect` был безопасен. У Next.js нет
аналогичной точки входа, которая гарантированно блокирует рендер клиентского
дерева до resolve какого-то промиса. Без этой гарантии `setLocale('en')`
вызывался бы до того, как словарь `en` зарегистрирован, и движок
(`LocaleRegistry.setLocale`) бросает `Error("No locales registered...")` —
краш всего приложения на клиенте.

Исправлено: `I18nProvider` теперь сам вызывает `await initI18n()` внутри
своего эффекта, и только потом `setLocale('en')`. Раньше это было неявным
контрактом ("кто-то снаружи обязан вызвать `initI18n()` до рендера"), теперь
это инкапсулировано внутри самого провайдера — он не зависит от того, что
происходит в точке входа приложения.

**Понятность для другого разработчика.** Стало понятнее, а не сложнее:
раньше нужно было знать про неявный порядок вызовов между `main.tsx` и
`I18nContext.tsx`, теперь `I18nProvider` самодостаточен — весь контракт
виден в одном файле.

**Нагрузка.** Не меняется — тот же один `fetch()` на языковой словарь при
монтировании, просто в другом месте кода.

**Расширяемость/заменяемость.** Легко: `initI18n()` идемпотентен по духу
(регистрирует локали в `Record`, повторный вызов просто перезапишет те же
ключи), так что дублирующий вызов ничего не ломает, если этот провайдер
позже обернут ещё раз выше по дереву.

**Известное ограничение (не исправлено сейчас, осознанно).** `ln()` —
словарные строки — недоступны до resolve `fetch()`, то есть в первый момент
рендера (в том числе при SSR на сервере, где `fetch()` в эффекте вообще не
выполняется) вызовы `ln("nav.work")` возвращают сам ключ как fallback
(поведение `LocaleRegistry.ln()` — `this.locales[this.currentLocale]?.[key] ?? key`),
и только после монтирования на клиенте подтягивается реальный перевод.
`pick()` (двуязычные литералы прямо в данных, например заголовки постов) эту
проблему не имеет — он не зависит от асинхронной загрузки вообще.

Проверено эмпирически при верификации Фазы 1 (`curl`/`Invoke-WebRequest` на
`npm run dev`, без выполнения JS): в HTML, отданном сервером ДО хидратации,
буквально виден текст `nav.work` вместо "Work" — то есть это не
гипотетический риск, а подтверждённое поведение. После хидратации на
клиенте `ln()` пересчитывается и текст должен самокорректироваться (тот же
`useEffect`, что чинит краш выше, вызывает `setLanguage('en')` уже ПОСЛЕ
`await initI18n()`, что триггерит ре-рендер всех потребителей контекста) —
но сам факт кратковременного "мигания" исходным ключом в разметке до
хидратации подтверждён, не предположение. План (Фаза 1) сознательно не
трогает архитектуру i18n сейчас — эта находка зафиксирована здесь, чтобы не
забыть про неё при более глубокой SSR-оптимизации позже (вероятное решение:
читать словарь `ln()` на сервере — например, из JSON-файла напрямую в
Server Component — вместо `fetch()` в клиентском эффекте).

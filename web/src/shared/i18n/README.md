# shared/i18n — перенос из frontend/, с найденным и по-настоящему исправленным FOUC-багом

Два независимых механизма живут рядом, как и в оригинале — `pick(value)`
для двуязычных объектов прямо в данных (`{en, ru}`, ничего не грузит) и
`ln("key")` для словарных строк UI-хрома (кнопки, лейблы навигации). С
2026-07-19 словари для `ln()` больше не грузятся по сети — см. запись
ниже "Настоящий фикс FOUC" — а вшиты в JS-бандл на этапе сборки
(`shared/i18n/locales/{en,ru}.json`, статический импорт).

## 2026-07-18 — Найденный краш при переносе на Next.js

**Что нужно сделать.** Перенести `I18nProvider` так, чтобы `ln()` не падал.

**Как сделано.** В старом Vite-приложении `main.tsx` делал `await initI18n()`
на уровне модуля **до** первого `ReactDOM.render(...)` — то есть к моменту,
когда `I18nProvider` монтировался, оба словаря уже были загружены, и
`setLocale('en')` внутри его `useEffect` был безопасен. У Next.js нет
аналогичной точки входа, которая гарантированно блокирует рендер клиентского
дерева до resolve какого-то промиса. Без этой гарантии `setLocale('en')`
вызывался бы до того, как словарь `en` зарегистрирован, и движок
(`LocaleRegistry.setLocale`) бросает `Error("No locales registered...")` —
краш всего приложения на клиенте.

Исправлено: `I18nProvider` теперь сам вызывает `await initI18n()` внутри
своего эффекта, и только потом `setLocale('en')`. Раньше это было неявным
контрактом ("кто-то снаружи обязан вызвать `initI18n()` до рендера"), теперь
это инкапсулировано внутри самого провайдера — он не зависит от того, что
происходит в точке входа приложения.

**Понятность для другого разработчика.** Стало понятнее, а не сложнее:
раньше нужно было знать про неявный порядок вызовов между `main.tsx` и
`I18nContext.tsx`, теперь `I18nProvider` самодостаточен — весь контракт
виден в одном файле.

**Нагрузка.** Не меняется — тот же один `fetch()` на языковой словарь при
монтировании, просто в другом месте кода.

**Расширяемость/заменяемость.** Легко: `initI18n()` идемпотентен по духу
(регистрирует локали в `Record`, повторный вызов просто перезапишет те же
ключи), так что дублирующий вызов ничего не ломает, если этот провайдер
позже обернут ещё раз выше по дереву.

**Известное ограничение (не исправлено сейчас, осознанно).** `ln()` —
словарные строки — недоступны до resolve `fetch()`, то есть в первый момент
рендера (в том числе при SSR на сервере, где `fetch()` в эффекте вообще не
выполняется) вызовы `ln("nav.work")` возвращают сам ключ как fallback
(поведение `LocaleRegistry.ln()` — `this.locales[this.currentLocale]?.[key] ?? key`),
и только после монтирования на клиенте подтягивается реальный перевод.
`pick()` (двуязычные литералы прямо в данных, например заголовки постов) эту
проблему не имеет — он не зависит от асинхронной загрузки вообще.

Проверено эмпирически при верификации Фазы 1 (`curl`/`Invoke-WebRequest` на
`npm run dev`, без выполнения JS): в HTML, отданном сервером ДО хидратации,
буквально виден текст `nav.work` вместо "Work" — то есть это не
гипотетический риск, а подтверждённое поведение. После хидратации на
клиенте `ln()` пересчитывается и текст должен самокорректироваться (тот же
`useEffect`, что чинит краш выше, вызывает `setLanguage('en')` уже ПОСЛЕ
`await initI18n()`, что триггерит ре-рендер всех потребителей контекста) —
но сам факт кратковременного "мигания" исходным ключом в разметке до
хидратации подтверждён, не предположение. План (Фаза 1) сознательно не
трогает архитектуру i18n сейчас — эта находка зафиксирована здесь, чтобы не
забыть про неё при более глубокой SSR-оптимизации позже.

## 2026-07-19 — Настоящий фикс FOUC (не смягчение — устранение причины)

**Что нужно сделать.** Пользователь вживую проверил сайт и увидел ровно
то, что было предсказано выше: при первом открытии сайта абсолютно ВСЕ
`ln()`-строки (пункты навигации, лейблы кнопок, статус) показывали сырые
ключи локализации, а не перевод — и это было заметно, не на долю секунды.

**Как сделано.** Разобрался, почему "известное ограничение" оказалось
хуже, чем "лёгкое мигание": `registerLocale("en")` и `registerLocale("ru")`
вызывались ПОСЛЕДОВАТЕЛЬНО (`await` одно, потом другое) внутри
`useEffect`, который сам по себе выполняется только на клиенте, только
после маунта, и только после того, как браузер вообще получил и
распарсил JS-бандл — то есть до того момента `ln()` в принципе не может
вернуть перевод, независимо от скорости сети. На первом холодном открытии
(без кеша браузера) это легко растягивается на сотни миллисекунд —
достаточно, чтобы увидеть, а не просто "мигание".

Вместо того чтобы просто ускорить/распараллелить `fetch()` (что уменьшило
бы задержку, но не убрало бы её — рендер до эффекта всё равно был бы без
перевода), устранена сама причина: два словаря — это два маленьких,
известных на этапе сборки JSON-файла для сайта с двумя языками. Не нужен
ни `fetch()`, ни `LocaleStore`-абстракция под подключаемые источники
загрузки, ни асинхронная инициализация вообще:

1. `shared/i18n/locales/{en,ru}.json` — словари, физически перенесённые
   из `public/locales/` (там больше не нужны — не отдаются по HTTP).
2. `LocaleRegistry` теперь принимает готовые словари в конструктор и
   работает полностью синхронно — убран `LocaleStore`/`WebLocaleStore`
   (удалены).
3. `engine/index.ts` создаёт `i18nEngine` прямо при загрузке модуля,
   через статический `import en from "../locales/en.json"` — движок
   готов ДО того, как отрендерится хоть один компонент, что на сервере,
   что на клиенте.
4. `I18nContext.tsx` — убран `useEffect`/`initI18n()` целиком.
   `I18nProvider` больше не асинхронный по духу вообще.

**Понятность для другого разработчика.** Раньше нужно было понимать
жизненный цикл эффекта, порядок `await`, отличие сервера от клиента, чтобы
понять, когда `ln()` "готов". Теперь `ln()` готов всегда — с первой же
строчки кода, который его вызывает. Меньше степеней свободы для новой
похожей ошибки.

**Нагрузка.** Строго лучше: ноль сетевых запросов на инициализацию (было
два последовательных `fetch()`), словари уже часть JS-бандла, который
браузер и так должен скачать.

**Расширяемость/заменяемость.** Третий язык — это третий JSON-файл и одна
строка в объекте `{en, ru, ...}` в `engine/index.ts`. Асинхронная
загрузка (например, если словари станут огромными и захочется грузить их
по требованию) — законный будущий шаг, но тогда это осознанное решение
под конкретную новую проблему (размер бандла), а не наследие с самого
начала.

**Миграция/отказоустойчивость.** Не применимо к данным — это код,
известный на этапе сборки.

**SOLID.** Single Responsibility стал чище: `LocaleRegistry` отвечает
только за "выбрать текущий язык и подставить строку", а не ещё и за
асинхронную оркестрацию загрузки, которая ему на самом деле была не
нужна для двух маленьких статических файлов.

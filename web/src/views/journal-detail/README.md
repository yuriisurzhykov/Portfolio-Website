## 2026-07-18 — useParams + редирект "не найдено" на Next.js API

**Что нужно сделать.** react-router даёт `useParams()` (синхронно читает
параметр маршрута) и `<Navigate to="..." replace/>` (декларативный редирект
прямо в JSX при рендере). У Next.js свой `useParams()` (похожий, из
`next/navigation`), но нет прямого аналога `<Navigate/>` для Client
Component — редирект из клиентского компонента можно делать только из
эффекта/хендлера через `useRouter().replace(...)`, не во время рендера.

**Как сделано.** `useParams<{slug}>()` заменён на `next/navigation`-версию
(API почти идентичен). Условие "пост не найден или ещё без текста" теперь
не рендерит `<Navigate/>` напрямую, а вызывает `router.replace("/journal")`
внутри `useEffect`, и в это же время (пока эффект не выполнился) компонент
рендерит `null`.

**Почему компонент остаётся Client Component, а не Server Component.**
`useTranslation()` — это React Context (`useContext` внутри) для `pick()` и
`ln()`, а Context недоступен в Server Components вообще. Раз план (Фаза 1)
сознательно не трогает архитектуру i18n сейчас, страница остаётся клиентской
— то есть настоящего SSR для текста этой страницы в Фазе 1 ещё нет
(HTML на сервере рендерится, но с пустым/неполным контентом до хидратации,
если контент зависит от i18n-текста через `ln()`; `pick()`-контент такой
проблемы не имеет, см. `shared/i18n/README.md`). Это сознательный,
задокументированный шаг — полноценный SSR для i18n-текста — предмет
отдельной, более глубокой работы, не Фазы 1.

**Понятность для другого разработчика.** Комментарий в коде прямо объясняет,
почему тут `useEffect`, а не `<Navigate/>`-подобный однострочник.

**Нагрузка.** Один лишний рендер-цикл (`null` → редирект) для несуществующего
slug — событие редкое (битая ссылка/черновик без `body`), не влияет на
обычный путь пользователя.

**Расширяемость/заменяемость.** Как только контент переедет в БД (Фаза 3),
"не найдено" естественно превратится в серверную проверку
(`if (!post) notFound()` в Server Component) — сегодняшний клиентский
редирект — временное решение под текущий источник данных (статический
`journal.ts`), не постоянная архитектура.

**Миграция/отказоустойчивость.** Не применимо к этому шагу — данные всё ещё
статические.

**SOLID.** Открыт вопрос Single Responsibility: страница одновременно ищет
пост, решает редирект и рендерит — приемлемо для Фазы 1 (компонент небольшой),
но при переезде на Server Components (Фаза 3) поиск/redirect естественно
переместится в `page.tsx` (маршрут), а этот компонент останется чисто
презентационным.

## 2026-07-19 — Новый слайс: перевод как отдельный экран (+ условная секция кейс-стади)

**Что нужно сделать.** См. `admin-post-translate/README.md`'s одноимённая
запись. Дополнительно для `Work`: перевод кейс-стади имеет смысл только
если у элемента вообще ЕСТЬ английский кейс-стади — иначе просто нечего
переводить.

**Как сделано.** `translation.hasCaseStudy` (из
`getWorkTranslationForAdmin`, backend) управляет видимостью всего блока
"Case study body" целиком — тот же принцип "один флаг → видимость целого
раздела", что уже применялся в `WorkEditorPage`'s `form.hasCaseStudy`
чекбоксе, только здесь это не чекбокс (перевод не решает, ЕСТЬ ли
кейс-стади — это решает английская версия), а просто факт, отражённый
текстом ("This item has no English case study yet"), когда раздела нет.

`summary` — единственное поле, которое переводится всегда, независимо от
`hasCaseStudy`: у любого `Work`-элемента есть `summary`, кейс-стади —
опционален.

`handleSubmit` всё равно отправляет `startedLabel`/`shippedLabel`/`role`/
`blocks` из формы, даже когда `hasCaseStudy` false (там просто лежат
значения из пустой `toFormState`) — backend (`translateWork`) сам решает
не трогать эти поля вообще, если `existing.caseStudyDocumentId` пуст (см.
его комментарий) — страница не обязана дублировать эту проверку перед
отправкой, лишняя пустая мутация на несуществующий кейс-стади безопасно
игнорируется на сервере.

**SOLID.** Interface Segregation: `translateWorkInputSchema` принимает
все четыре кейс-стади-поля как плоские (не завёрнутые в
`caseStudy: {...} | null`, в отличие от `workInputSchema`) — на этой
странице нет чекбокса "есть кейс-стади", поэтому нет и причины моделировать
это как выбор между двумя формами полей.

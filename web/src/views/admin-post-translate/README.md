# views/admin-post-translate — PostTranslatePage

## Будет ли переиспользоваться?

Внутри этой страницы — да, оба поля-с-подсказкой (`ReferenceField`)
переиспользуются между title/category/excerpt, а не написаны по отдельности
три раза. Сама страница переиспользуется в другом смысле: она — точное
структурное отражение `admin-work-translate`'s `WorkTranslatePage`, то же
решение ("перевод — отдельная страница, не поле рядом"), применённое к
`Post` вместо `Work`.

## 2026-07-19 — Новый слайс: перевод как отдельный экран

**Что нужно сделать.** По плану, английская версия поста — обязательный,
основной контент; перевод — необязательное отдельное действие. Раньше
(до этого плана) `PostEditorPage` держал EN и RU рядом в одной форме
(`LocalizedInputField`) — здесь эта форма разделяется на два раздельных
экрана с раздельными API-вызовами.

**Как сделано.** `PostTranslatePage({ translation: AdminPostTranslation })`
— `translation` НИКОГДА не отсутствует (в отличие от `PostEditorPage`'s
`initialPost?`): "перевести" не бывает для ещё не созданного поста, маршрут
(`app/admin/(dashboard)/journal/[slug]/translate/page.tsx`) вызывает
`notFound()` раньше, если `slug` не существует вообще. Каждое поле —
`ReferenceField` (label + `hint` с английским оригиналом в кавычках) вокруг
обычного `Input`/`Textarea`, редактирующего ТОЛЬКО `ru`-половину — не
`LocalizedInputField` (два независимых инпута рядом), потому что здесь
редактируется только одна сторона, вторая — только для чтения, показанная
текстом, не полем.

`handleSubmit` отправляет `TranslatePostInput` (`title`/`category`/
`excerpt`/`blocks` — только эти четыре поля, ничего похожего на
`PostInput`'s `slug`/`date`/`status`/...) через `adminApi.translatePost`
(`PUT /api/admin/posts/:slug/translation`) — отдельный вызов от
`updatePost`, физически не способный тронуть английскую версию или
scalar-метаданные поста.

Блочный редактор — тот же `<BlockEditor>` (`shared/ui/block-editor`), что
и в `PostEditorPage`, с `initialBlocks={translation.blocks}` — русские
блоки тела (`[]`, если перевода ещё нет — см. `getPostTranslationForAdmin`'s
комментарий в backend про то, почему это НЕ фолбэк на английские блоки).

**Понятность для другого разработчика.** Название `ReferenceField` прямо
говорит, что это "поле + для справки" — не "локализованное поле", чтобы
не путать с удалённым `LocalizedInputField`, который решал похожую, но не
идентичную задачу (два РЕДАКТИРУЕМЫХ поля, не одно редактируемое + одно
только для чтения).

**Нагрузка.** Не применимо — обычная клиентская форма.

**Расширяемость/заменяемость.** Третий язык (если понадобится) — новый
слайс `admin-post-translate-<lang>` с тем же паттерном, не правка этого
файла — перевод целиком независим от того, сколько других переводов уже
существует.

**Миграция/отказоустойчивость.** Не применимо.

**SOLID.** Single Responsibility, доведённый до предела: эта страница не
знает НИЧЕГО про английский контент, кроме как показать его читателю —
не может его случайно перезаписать, потому что у неё просто нет мутации,
которая касается `en`-полей.

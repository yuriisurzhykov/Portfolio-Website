# widgets/layout — AppLayout

## 2026-07-18 — Перенос с react-router на Next.js App Router

**Что нужно сделать.** `AppLayout` больше не может оборачивать `<Outlet/>`
(react-router) — в App Router контент передаётся как `children`, а сама
разметка (`<Nav/>` + контент + `<Footer/>`) подключается через
`app/layout.tsx`.

**Как сделано.** Сигнатура изменена с `AppLayout()` (без пропсов, читает
маршрут через контекст react-router) на `AppLayout({ children })`. Логика
"проскроллить к якорю или в начало страницы при смене маршрута" переписана:
было `useLocation()` (pathname + hash одним объектом), стало
`usePathname()` из `next/navigation` для смены маршрута +
`window.location.hash`, прочитанный вручную в эффекте — Next.js
принципиально не отдаёт hash через свои роутинг-хуки (это чисто
браузерное понятие, сервер о нём не знает).

**Понятность для другого разработчика.** Комментарий в коде явно объясняет,
почему `usePathname()` не заменяет `useLocation()` 1:1 — если бы этого
комментария не было, было бы неочевидно, почему hash читается отдельно.

**Нагрузка.** Не меняется — тот же один `useEffect` на смену маршрута.

**Расширяемость/заменяемость.** Компонент принимает `children`, что даёт
больше свободы, чем `Outlet` — в будущем этот же `AppLayout` можно
использовать и там, где раньше он был жёстко завязан на react-router route
tree (например, в тестовом рендере страницы в изоляции).

**Миграция/отказоустойчивость.** Не применимо (нет данных).

**SOLID.** Single Responsibility не пострадал — компонент как и раньше
отвечает только за "рамку" страницы (nav/footer/scroll), не за то, что
внутри неё рендерится.

## 2026-07-19 — Подключена `<BackToTop/>`

Добавлена рядом с `<Nav/>`/`<Footer/>` как третий постоянный элемент
"рамки" страницы. Сам компонент — `fixed`, поэтому его позиция в дереве
между `<Footer/>` и закрывающим `</div>` не влияет на раскладку; выбрано
это место просто чтобы держать все элементы уровня "site chrome" вместе в
JSX. Обоснование и связь с `<Nav/>` — в `shared/ui/back-to-top/README.md`.

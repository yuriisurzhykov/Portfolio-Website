# widgets/nav — Nav

## 2026-07-18 — Перенос с react-router на Next.js App Router

**Что нужно сделать.** `NavLink` из react-router-dom умеет сам подсвечивать
активный пункт меню (`({isActive}) => ...`) и знает про hash из
`useLocation()`. В Next.js этого готового компонента нет.

**Как сделано.** `NavLink` заменён на обычный `next/link`'а `Link` +
самостоятельное сравнение `usePathname() === "/work"` для активного пункта.
Для пункта "Contact" (`/#contact`) используется тот же приём с
`window.location.hash` + слушателем `hashchange`, что и в `AppLayout` (см.
его README) — вынесено в локальный хук `useUrlHash()` в этом же файле, а не
переиспользуется оттуда, так как оба места самодостаточны и невелики
(YAGNI — не создавать общую абстракцию для двух мест, пока не появится
третье).

**Понятность для другого разработчика.** Явный `navLinkClass(isActive: boolean)`
вместо неявной API `NavLink`, читается прямо в JSX, что именно сравнивается
с чем.

**Нагрузка.** `hashchange`-listener добавляется один раз на монтирование
`Nav`, снимается при размонтировании — обычной памяти на страницу с одним
`Nav` это не добавляет ничего ощутимого.

**Расширяемость/заменяемость.** Если позже появится третье место, которому
нужен `hash`, стоит вынести `useUrlHash()` в `shared/lib` — сейчас
преждевременно.

**Миграция/отказоустойчивость.** Не применимо.

**SOLID.** Компонент остаётся ответственным только за навигацию/шапку —
получение состояния urlhash поведенчески инкапсулировано в отдельный хук
внутри файла, а не размазано по JSX.

## 2026-07-19 — Мобильное меню (найдено пользователем на реальном скриншоте)

**Что нужно сделать.** На мобильном экране шапка визуально шире, чем
контентная колонка ниже — логотип, три пункта навигации, переключатель
языка, переключатель темы и статус-бэйдж не помещались в одну строку и
переносились некрасиво.

**Как сделано.** Ниже breakpoint `sm` шапка теперь показывает только
логотип и кнопку-гамбургер (`sm:hidden` на кнопке, `hidden sm:flex` на
исходном ряду ссылок и переключателей — сам ряд не удалён, просто скрыт на
узких экранах, `sm:` и выше выглядят ровно как раньше). Клик по гамбургеру
открывает `<Drawer/>` (`shared/ui/drawer`, общий компонент — см. его
README про то, почему он не специфичен для `Nav`) со своей копией
ссылок/переключателей, свёрстанной вертикально для мобильного экрана.

Меню закрывается: по клику на "×", по клику на ссылку внутри (переход +
закрытие одним кликом, не два отдельных действия), по Escape, по клику на
подложку (всё — механика `Drawer`), и дополнительно — при смене маршрута
(`useEffect` на `pathname`, на случай навигации каким-то другим путём,
пока меню было открыто).

**Понятность для другого разработчика.** Десктопный и мобильный варианты
ссылок — раздельные JSX-блоки (`navLinks`/`mobileNavLinkClass`), а не один
набор ссылок, стилизуемый по-разному через классы — на мобильном экране
нужны более крупные touch-target'ы (`text-body-lg py-sm` вместо
`text-caption`), а не просто увеличенный текст того же элемента.

**Нагрузка.** Панель существует в DOM даже когда закрыта (см. `Drawer`
README про `inert`), но это статичная разметка без собственного состояния
— пренебрежимо.

**Расширяемость/заменяемость.** Контент мобильного меню (ссылки +
переключатели + бэйдж) — тот же набор данных, что и в десктопной шапке,
просто свёрстанный иначе; при добавлении нового пункта навигации нужно
помнить добавить его в оба места (осознанный trade-off простоты вместо
абстрагирования "списка пунктов меню", который сейчас избыточен для трёх
статичных ссылок).

**Миграция/отказоустойчивость.** Не применимо.

**SOLID.** Open/Closed в лёгкой форме: логика открытия/закрытия/фокуса
целиком в `Drawer`, `Nav` только предоставляет содержимое — расширение
поведения самой панели (например, анимация появления другого типа) не
требует трогать `Nav.tsx`.

## 2026-07-19 — Скрытие шапки при скролле вниз, появление при скролле вверх

**Что нужно сделать.** Пользователь: шапка `sticky` должна прятаться при
скролле вниз (не занимать место над контентом всё время), но снова
появляться при скролле вверх — а не просто быть статично прилипшей к верху
всегда.

**Как сделано.** Условие "нужно ли прятать шапку" вынесено в
`shared/lib/useHideOnScroll()` — не локальный хук в этом файле, в отличие
от `useUrlHash` выше, потому что оно используется ещё в одном месте
(`shared/ui/back-to-top` — плавающая кнопка "наверх", появляющаяся ровно
тогда, когда шапка прячется; подробности и находки при тестировании — в
её README). К существующим классам `sticky top-0 z-navbar` добавлен
`transition-transform` + условный `-translate-y-full`, когда хук вернул
`true`. `position: sticky` не заменён на `fixed` — трансформ не влияет на
layout, поэтому шапка так же "прилипает" к верху при скролле, просто
дополнительно уезжает за пределы экрана, когда невидима.

**Понятность для другого разработчика.** Комментарий в JSX явно указывает,
что `<BackToTop/>` в `AppLayout` использует тот же хук с тем же порогом —
без этой связи неочевидно, почему две с виду независимые части интерфейса
переключаются синхронно.

**Нагрузка.** Один `scroll`-listener на `Nav`, throttled через
`requestAnimationFrame` — не больше одного пересчёта состояния за кадр.

**Расширяемость/заменяемость.** Порог, при котором шапка ещё считается "у
hero" (не прячется, даже если скроллят вниз), — параметр хука, не magic
number внутри `Nav.tsx`.

**Миграция/отказоустойчивость.** Не применимо.

**SOLID.** Single Responsibility не пострадал: `Nav` не считает
направление скролла сама, только применяет готовый булев результат к
своим классам.

## 2026-07-19 — `LanguageSegmentedToggle`: с клиентской кнопки на настоящие ссылки

**Что нужно сделать.** Язык теперь часть URL (`/ru`-префикс, `proxy.ts`
rewrite) — переключатель, который просто вызывал `setLanguage()` на
клиентском контексте, больше не может быть источником истины: сервер про
такое переключение никогда не узнал бы.

**Как сделано.** `LanguageSegmentedToggle` (`widgets/nav/LanguageSegmentedToggle.tsx`)
теперь рендерит два `<a href=...>` — не `next/link`'а `<Link>`, специально
обычный `<a>`. `next/link` дал бы клиентскую soft-навигацию, при которой
Next.js не гарантирует повторный вызов `RootLayout` (общий layout, где
`getRequestLocale()` резолвит `initialLanguage` один раз на запрос) —
обычная браузерная навигация (`<a>`) заново проходит через `proxy.ts`
и весь дерево рендерится с нуля, что для смены языка (редкое,
осознанное действие пользователя) — приемлемая цена за гарантированную
корректность, а не оптимизация, которую стоило чинить сложнее.

`hrefFor()`/`stripRuPrefix()` — тот же алгоритм, что в `proxy.ts`'s
`handleLocale` (снять/добавить `/ru`), продублированный намеренно (два
маленьких, самодостаточных места), а не вынесенный в общий модуль —
`proxy.ts` работает с `NextRequest`/`NextResponse.rewrite`, этот компонент
— с `usePathname()`; общая абстракция сейчас связала бы Edge-код и
клиентский компонент ради пары строк.

**SOLID.** Не изменилось структурно — компонент всё еще не знает ничего
про то, ЧТО происходит на сервере при переходе по этой ссылке, только
формирует правильный `href`.

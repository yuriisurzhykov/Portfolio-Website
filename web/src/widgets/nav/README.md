# widgets/nav — Nav

## 2026-07-18 — Перенос с react-router на Next.js App Router

**Что нужно сделать.** `NavLink` из react-router-dom умеет сам подсвечивать
активный пункт меню (`({isActive}) => ...`) и знает про hash из
`useLocation()`. В Next.js этого готового компонента нет.

**Как сделано.** `NavLink` заменён на обычный `next/link`'а `Link` +
самостоятельное сравнение `usePathname() === "/work"` для активного пункта.
Для пункта "Contact" (`/#contact`) используется тот же приём с
`window.location.hash` + слушателем `hashchange`, что и в `AppLayout` (см.
его README) — вынесено в локальный хук `useUrlHash()` в этом же файле, а не
переиспользуется оттуда, так как оба места самодостаточны и невелики
(YAGNI — не создавать общую абстракцию для двух мест, пока не появится
третье).

**Понятность для другого разработчика.** Явный `navLinkClass(isActive: boolean)`
вместо неявной API `NavLink`, читается прямо в JSX, что именно сравнивается
с чем.

**Нагрузка.** `hashchange`-listener добавляется один раз на монтирование
`Nav`, снимается при размонтировании — обычной памяти на страницу с одним
`Nav` это не добавляет ничего ощутимого.

**Расширяемость/заменяемость.** Если позже появится третье место, которому
нужен `hash`, стоит вынести `useUrlHash()` в `shared/lib` — сейчас
преждевременно.

**Миграция/отказоустойчивость.** Не применимо.

**SOLID.** Компонент остаётся ответственным только за навигацию/шапку —
получение состояния urlhash поведенчески инкапсулировано в отдельный хук
внутри файла, а не размазано по JSX.

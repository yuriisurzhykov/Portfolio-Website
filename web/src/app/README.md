# app — маршрутизация (Next.js App Router)

Заменяет `frontend/src/app/router/routes.tsx` (react-router `<Routes>`)
файловой системой маршрутов Next.js. `app/providers` и `app/styles`
перенесены из `frontend/src/app` почти без изменений (см. их README, если
логика внутри менялась).

## 2026-07-18 — Route group `(site)` для сохранения "no chrome" у /storybook

**Что нужно сделать.** В исходном роутере `/storybook` был явным исключением
— единственный маршрут, который НЕ оборачивался в `<AppLayout/>`
(без `<Nav/>`/`<Footer/>`), потому что это dev-only витрина дизайн-системы,
не часть публичного сайта.

**Как сделано.** Next.js Route Groups: `app/(site)/` — папка в круглых
скобках, не влияет на URL, но имеет свой `layout.tsx`, оборачивающий детей в
`<AppLayout/>` (Nav/Footer). Все публичные страницы (`/`, `/work`,
`/work/[slug]`, `/journal`, `/journal/[slug]`) лежат внутри неё.
`app/storybook/page.tsx` — вне этой группы, получает только корневой
`app/layout.tsx` (html/body/providers/design-tokens), без Nav/Footer —
эквивалент исходного поведения.

**Понятность для другого разработчика.** Название `(site)` прямо говорит
"это публичный сайт с обычной навигацией", а расположение `storybook/` вне
скобок — что это осознанное исключение, а не забытый файл.

**Нагрузка.** Не применимо — организация маршрутов, не рантайм-логика.

**Расширяемость/заменяемость.** Если позже появится третий "профиль"
разметки (например, `/admin` со своей навигацией, Фаза 4 плана), это будет
ещё одна route group на этом же уровне — паттерн уже установлен.

**Миграция/отказоустойчивость.** Не применимо.

**SOLID.** Interface Segregation в некотором смысле: каждая route group
предоставляет только ту "рамку", которая нужна её маршрутам — publicному
сайту не навязывается ничего лишнего, и наоборот.
